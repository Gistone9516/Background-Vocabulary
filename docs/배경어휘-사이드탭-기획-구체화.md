### 1-1. 존재이유 방어 (가장 중요, deepflow C1에서 격상)

비판 패널 세 렌즈(사후부검, 제1원리, UX)가 공통으로 친 급소: **"메인 AI에게 그냥 물어보면 되는데 왜 이 탭을 먼저 여나?"** 이건 열어둬도 되는 질문(원래 §9)이 아니라 제품 생사를 가르는 전제이므로 여기서 답을 못 박는다.

- **방어 명제:** 메인 AI의 되묻기(clarifying question)는 사용자가 *무엇을 물어야 할지 어느 정도 아는* 상태를 전제로 작동한다. 이 서비스는 **"무엇을 모르는지조차 모르는" 상태**에서 출발해, 분야 어휘 지형을 비선형으로 펼쳐 클릭만으로 의중을 외화한다. 선형 대화로는 "내가 뭘 모르는지"가 드러나기 전까지 좁혀지지 않는다.
- **부차 차별:** 대화창 바깥의 독립 준비 공간이라는 위상, 어휘를 한눈에 보는 시각 탐색 경험.
- **종착점 교정 (2026-06-24 사용자 directive):** 이 제품의 종착점은 "복사용 프롬프트 생성"이 아니라 **사용자가 분야 개념을 이해해 스스로 말할 수 있게 되는 것**이다(핵심 표면 = 화면2 상세 열람·드롭다운). 복사·공유(화면3)는 그 이해를 선택적으로 가져가는 편의일 뿐 목적이 아니다. 측정 편의 때문에 paste를 성공 지표로 박았던 것이 정체성을 덮었던 것을 바로잡음(§7-1 반영).
- **이 명제는 아직 가정이다.** MVP에서 직접 검증한다(§7-1의 존재이유 가설). 깨지면 제품 방향 재고. 기획 단계인 지금은 검증하지 않고 검증 설계만 박아둔다.
- **흡수 리스크 동반:** 같은 공백을 메인 AI가 네이티브로 메우면 차별점이 소멸할 수 있음(§7-3 리스크).

> **DeepSeek-V4 미검증 항목 (착수 전 선결 확인):** 이번 선행조사는 단가·지연을 Haiku·Gemini·GPT 중심으로 측정했고, **DeepSeek-V4의 ① 실시간 단가 ② 한국 접속 기준 TTFT ③ 한국어 전문어휘 생성 품질 ④ 프롬프트 캐싱 지원 여부는 직접 검증하지 않았다.** 이 네 가지는 비용·UX·결과품질을 좌우하는 **착수 전 선결 확인 항목**이다(기획 단계인 지금은 검증하지 않음). 부정으로 판명될 때의 대체 모델 fallback은 §7-3에 둔다. (조사 단계에서 "DeepSeek V4-Flash 안정성 불명" 신호가 있었음)
>
> **[2026-06-22 실측 갱신]** 프롬프트3을 flash/pro로 실측(`정확도-최종-결과.md`): ① JSON mode 지원 확인 ② 한국어 전문어휘 품질 = pro 평균 ~77/100·flash ~67(고위험 도메인 오류·할루시네이션 잔존) ③ latency = pro ~51초·flash ~34초(thinking ON, 한국 TTFT 1.5초 우려 잔존) ④ **모델은 pro 채택**(out-of-sample 15% 기준은 pro만 충족). 단가 정밀·캐싱·대규모 부하는 여전히 0단계 선결. **공통 최신성 오류(간이과세 구기준 등)는 모델 컷오프 문제라 RAG·경고로 보완 필요(§7-3).** **[0단계 격자 실측]** thinking OFF는 TTFT 0.8초로 속도는 통과하나 정확도 급락(no-RAG flash 53·pro 59), **RAG 주입 시 +25~27점·최신성 100% 교정**. 정확도 85 목표 통과는 **pro + thinking OFF + RAG(85.4)** 뿐 → 채택 레시피(+ term 단위 스트리밍). flash+RAG(78)는 저위험 폴백. 상세 `0단계-실측-결과.md`.

## 2. 핵심 작동 루프 (MVP, 상세화)

원본 §7 루프를 화면 단위로 풀어쓰되 **입력 구조는 종합안(다지선다 엔진 + 어휘 태깅, 사용자 확정)**을 따른다. 각 화면의 실제 UI 문구 초안까지 박는다(구현자 임의결정 방지). 문구는 잠정안, 관찰로 다듬음.

> **입력 구조 결정 (사용자 확정):** 의중을 좁히는 본 루프는 **내용을 담은 3~4지선다**다. "어휘를 보고 아는가"가 아니라 "이 상황 중 무엇이 당신에 가까운가"를 고르게 한다. 재인(recognition) 기반이라 비전문가의 메타인지 부담이 작고, 정보이론 분기(원본 §4)와 정합한다. [알아/몰라]는 마지막에 **좁혀진 핵심어 묶음을 태깅**하는 보조 단계로만 쓴다. 근거는 §3.

> **MVP 범위 (작업한계 없음 반영):** 실시간 적응형 분기(매 선택마다 다음 선택지를 LLM이 생성)를 1차에 포함한다. 고정 선택지만으론 "클릭으로 의중이 좁혀지는가"를 검증 못 함. 실시간에 따르는 견고성 장치(JSON 계약 §5-1, pending 잠금 §3-1, 선택지 품질 가드·종료조건 §3-2, 지연 fallback §7-3)를 함께 구현. 1차는 LLM-prompted, 엔트로피 실제 계산(greedy EIG)은 2차.

## 3. 인터랙션 레이어 설계 (알맹이)

이 서비스의 재사용되는 핵심. 뒤에 붙는 게 사전이든 검색이든 이 층은 그대로다.

### 3-1. 상태 모델
- **의중 후보 분포**: 분야 안 세부 영역들에 대한 확률 가중치. 매 선택마다 갱신(좁히기 엔진의 핵심 상태).
- **사용자 지식 상태**: 각 어휘에 대해 알아/몰라/적용모름/미확인(4상태, P11). (조사 §2: knowledge tracing을 "정오답"이 아니라 "클릭 패턴"으로 재정의, 선행과 차별)
- **탐색 경로**: 지나온 선택지·어휘와 선택 이력 (정리·되돌리기·로그용).
- **맥락 객체(context_object, 시나리오 검증 §10):** 진입 시 붙여넣은 원문(오류 메시지·등기부 수치·논문 문단 등)을 분야 추론용으로만 쓰지 말고 보존해 **화면1 선택지 생성(프롬프트2)·화면2 어휘 추천(프롬프트3)·정리(프롬프트4)까지 끝까지 이월**한다(§5-1 계약에 입력 필드로 명시, P16). 추론용으로만 쓰고 버리면 화면1~2가 맥락 없이 진행돼 일반론만 나온다(P4·P16).
- **세션 간 이월 (개인화 1차 부분, deepflow C3):** 지식 상태를 localStorage에 영속화해 재방문 시 불러온다. 이미 [알아] 한 어휘는 덜 노출, [몰라] 한 어휘는 복습 우선. 방문마다 제로에서 시작하지 않게 해 존재이유(§1-1)를 물질화.
- **비동기 잠금(pending), UI 상태 명시 (deepflow C3):** LLM 호출(다음 선택지 생성, 정리 생성) 중에는 **이전 화면 내용은 그대로 두고 버튼만 비활성과 스피너로 교체**(레이아웃 점프 없음). 2초 초과 시 "다음 선택지 고르는 중..." 인라인 안내. 연속 클릭 경합과 빈 화면을 함께 막는다.

### 3-2. 선택지 생성 — 1차/2차 구분 + 운영 규칙
- **1차(MVP): LLM-prompted 선택지 생성 (EIG 모사).** 매 선택 누적을 LLM에 넣어 다음 3~4개 내용형 선택지를 실시간 생성하되, 엔트로피를 실제 계산하지 않고 "의중을 가장 크게 가르는 선택지를 내놓으라"고 지시하는 휴리스틱이다. EIG라는 이름을 붙이지 않는다(과장 방지). 1차에서 볼 것은 "LLM 선택지가 의중을 좁히는 데 쓸 만한가"이지 "EIG가 도는가"가 아니다.
- **2차: 진짜 greedy EIG.** 동작은 같되 누적 로그로 후보 분포 엔트로피를 *실제 계산*해 가장 크게 줄이는 분기를 고른다(LLM 휴리스틱에서 정보이론 최적화로). 1차와 2차 기준선을 여기서 분리.
- **수렴·종료 조건 (적응형 필수, deepflow C3):** 실시간 분기는 "선택지 소진"이 없으므로 시스템이 끝을 정해야 한다. (a) 누적 선택 N회 초과(예: 6~8), (b) [더 깊이] 누적 k회 도달(예: 3), (c) 같은 도메인 태그 연속 j회(예: 3) 중 하나면 자동으로 정리 화면 유도("이 정도면 충분해 보입니다, 정리 볼까요?"). 수치는 §7-2 관찰로 보정. ([지금 정리 보기]는 사용자가 언제든 누르는 탈출구이고, 자동 유도는 종료 조건 충족 시 그 버튼을 강조하는 것)
- **[더 깊이] depth cap (deepflow C3):** 세션 내 누적 상한(예: 3회). 도달 시 버튼 비활성화 + "정리 먼저 보고 새로 파보세요" 안내.
- **선택지 품질 가드:** (착수 전) **시나리오 대표 분야로 프롬프트1·2·3을 수동 평가**한다 — 35개 시나리오가 몰린 고난도 분야(부동산·임대차 법률, 세무·상속, 의료 검사 해석, 정부지원사업, API 명세, 보건역학) 중 5~6개를 골라 선택지뿐 아니라 **추천 어휘의 use_example·direction·relates_to 정확도까지** 점검(P24, 글쓰기·진단·판단 클러스터의 핵심 전달 수단이라 사전 검증 필수). (런타임) 직전 2개 선택지와 신규의 domain_tags 교집합이 절반 이상이면 재생성(최대 2회), 그래도 겹치면 다른 태그를 강제 지정한 선택지로 fallback.
- **공통 고정구간 (deepflow C3):** 같은 분야 첫 1~2 분기 선택지는 모든 사용자 동일(프롬프트1 캐시). 5명 관찰 비교가능성 확보(§7-2). 적응은 그다음부터.
- noisy 입력 대비: 되돌리기 1단계 + 확률적 갱신(한 번 선택으로 후보를 0으로 죽이지 않음).
- **선택지 레이블 = 일상어 상황문 (P6, 자기참조 함정 차단):** 레이블은 그 분야를 처음 접한 사람이 뜻을 몰라도 자기 상황에 해당하는지 판단할 수 있는 일상어로 쓴다. 분야 전문어는 레이블이 아니라 domain_tags에만(어휘를 몰라 선택지조차 못 고르는 IT·법률·통계 계열에서 임계적).
- **작업유형 고정 분기 (P2·P18·P25):** 첫 1~2 분기 중 하나는 작업 유형을 가르는 분기로 고정 배치해 의중의 닻을 일찍 잡는다. 화면1 버튼(일상어)과 내부 job_type 열거값의 매핑을 못 박는다(구현자 자의 금지, P25):
   - "글·문서 쓰기" 다음 보고서작성, 기획전략
   - "결정·비교하기" 다음 의사결정
   - "서류·신청서 채우기" 다음 서류제출
   - "이해·준비하기" 다음 이해학습, 전문가면담준비
   - "협상·설득 준비" 다음 협상설득준비
   - "따져보기·진단" 다음 진단판단, 문제해결

   한 입력이 두 작업을 겸하면(예: 이해+글쓰기, 진단+협상) job_type을 최대 2개까지 잡아 정리·추천에 함께 반영한다(복수 job_type, P18). 자유문장으로 들어와 버튼을 안 거치면 프롬프트1이 위 매핑 기준으로 job_type을 직접 부여한다(화면1 버튼과 동일 기준이라 일관 보장).
- **fast-track (P5·P20):** 입력에 비교 대상 2개 이상, 구체 조항명·수치, 또는 자유문장에서 모르는 어휘·항목을 이미 구체적으로 지목한 경우 의중이 이미 구체화된 것으로 보고 좁히기를 1~2 라운드로 단축하거나 건너뛰고 어휘 태깅으로 직행.
- **복수 의중 허용 (P10):** 첫 분기에서 의중이 두 갈래 이상으로 감지되면(domain_tags 복수) 복수 선택(최대 2개)을 허용하고 두 갈래를 정리에 함께 담는다.
- **배경 수준 포착 (P8):** 기본은 프롬프트1이 입력에서 수동 추론(P15 비전공자 기본값). 추론이 모호할 때만 좁히기 중 1회 능동 질문("처음 접하나 / 현장 경험은 있는데 공식 언어가 낯선가 / 이전에 배웠는데 다시 정리하나")으로 보강해 context_sentence에 반영.
- **조건 조기 수집 — fast-input (P16·P17, 시점 역전 해소):** user_condition·context_object는 화면2 추천 시점에 있어야 priority가 개인화된다. 그래서 ① 화면0 보조 입력(분야명·자유문장 외에 "관련 조건을 한 줄로" 선택 입력)과 ② 붙여넣은 원문(context_object)을 화면0~1에서 받아 프롬프트2·3에 함께 전달한다. [내 상황 추가](화면3)는 마지막 보강 채널로 유지하되, 추천 개인화의 1차 입력은 화면0~1에서 받는다. §3-1 context_object 이월 선언과 §5-1 계약을 여기서 일치시킨다.
- **조건 필수 판단 (P17·P29, 조건 의존도 기반):** 고정된 도메인 목록이 아니라 "user_condition 없이는 추천·정리가 일반론에 그치는가"라는 **조건 의존도**로 판단한다(목록 하드코딩은 커버리지 누락을 부른다, P29). 프롬프트1이 의존도가 높다고 보면 화면0~1 조건 입력을 "선택"에서 "권장"으로 승격하고 케이스별 핵심 변수를 힌트로 제시한다(예: 거주 기간, 보유 수치, 소득, 건물 속성, 공시 범위, 검사 항목). 조건 의존 高 예시: 비교·판단·진단·세무·상속·법률·의료·ESG 공시·교육 행정·건축 행정·사회보험. **이 예시는 프롬프트 설계 시 few-shot 참고로만 쓰고 시스템 프롬프트에 화이트리스트로 나열하지 않는다**(나열하면 목록 외 조건 의존 도메인이 누락됨, P30). 판정 결과는 프롬프트1이 condition_required(bool)로 출력하고 프론트가 그 플래그로 승격한다(§5-1). 입력 마찰과 정확도의 트레이드오프는 §7-2에서 관찰한다.
- **user_condition 병합·재추천 규칙 (P26):** user_condition은 두 채널에서 들어온다 — 화면0~1 보조입력(조기, 추천 개인화용)과 화면3 [내 상황 추가](보강·확정). **병합 우선순위: 화면3 최종값이 화면0 값을 덮는다**(마지막에 확인·수정한 값이 진실). 화면0 값은 화면3에 prefill로 보여 보강만 받고, 프롬프트4에는 이 병합 최종값을 넣는다. **재추천 트리거:** 조건을 화면0~1에서 못 받고 화면3에서 처음 받았거나 핵심 변수가 크게 바뀌면 화면2 추천은 이미 일반론으로 생성된 상태다. 이때 자동 재호출하지 않고(비용·지연) "바뀐 상황으로 추천 다시 볼래요?" 버튼을 제시해 누르면 프롬프트3을 재호출, 안 누르면 paste_text(프롬프트4)에만 반영한다.

### 3-3. 데이터 단위 (선택지 / 어휘 카드)
좁히기 엔진이 다루는 **선택지(Choice)**와 마지막 태깅 단계의 **어휘 카드(Term)**를 구분한다. JSON 계약은 §5-1.
```
Choice {                          // 화면1, 다지선다 한 항목
  label:       "가격이 왜 그렇게 매겨지는지"   // 일상어 상황문 필수(P6). 전문어 금지
  domain_tags: [미시경제, 가격이론]            // 전문어는 여기에만. 의중 분포·중복 가드용
  deeper:      true                            // [이 갈래 더 깊이] 가능 여부
}

Term {                            // 화면2, 우선순위 추천 어휘 카드(P13)
  term:         "한계효용"
  kind:         개념 | 방법론 | 지표 | 제도·규정 | 도구 | 절차 | 역할   // 단어의 종류(P14)
  group:        "학습 안정화"                  // 상위 주제(클러스터). 그룹핑·지도 보기 조직 단위. 필수(P32)
  priority:     1                              // 이 상황 적합 우선순위(추천 핵심)
  why:          "가격 근거를 따지려는 당신 목적에 직접 쓰임"  // 왜 우선인지, 근거 필수
  one_line:     "한 단위 더 소비할 때 늘어나는 만족"
  tag:          미확인 | 알아 | 몰라 | 적용모름   // 알아=추천서 빼기, 몰라/적용모름=집중
  direction:    "수치 높을수록 유리"            // 선택, 판단·협상·제도 맥락(P7)
  use_example:  "가격 인하 정당화: 한계효용 체감으로 추가 할인의 매출 증가폭이 준다"  // 선택, 글쓰기(P12)
  context_note: "당신 상황에선 이렇게 작동"      // 선택, 적용 한 줄(P11)
  relates_to:   ["고정비"]                       // 선택, vocab_gap 유형 b. 이 어휘와 묶어 봐야 하는 다른 term (관계 표현, P19)
  order:        2                                // 선택, 유형 b. 개념·절차의 선후 순서 (작으면 먼저, P19)
  gap_type:     ["c"]                            // 선택, 이 어휘의 막힘 유형. 세션 기본값을 어휘별로 오버라이드(P27). a~e, 복합이면 배열
}
```
- **vocab_gap 유형 분기(P7):** 막힘은 (a)뜻 모름 (b)개념 관계·순서 (c)재번역(기존 언어에서 전문 언어로) (d)수치·결과 해석 (e)조항 상호작용으로 나뉜다. **유형 다음 Term 필드 매핑(고정, P27): a는 one_line, b는 relates_to·order, c는 use_example(재번역 예시)와 direction, d는 direction·context_note(수치 해석), e는 context_note·direction(상호작용·유불리).** "연결 문장"을 why·context_note에 비정형으로 구겨넣지 않는다(P19). **Term.gap_type가 있으면 그 어휘는 Term.gap_type만 적용한다(세션 기본값과 합집합이 아니라 대체, P30).**
- **gap_type 결정 주체·시점 (P27, 닭-달걀 해소):** gap_type은 화면2 추천(프롬프트3)의 입력이라 그 전에 정해져야 한다. 결정 흐름은 **프롬프트1·2가 진입 자유문장·선택 이력·context_object에서 session 기본 gap_type을 LLM 추론**해 프롬프트3에 넘긴다(사용자 [몰라/적용모름] 태깅은 화면2 이후라 늦으므로 추론이 1차). 어휘마다 막힘이 다르면 Term.gap_type로 어휘별 오버라이드(세션 기본값 위에 얹음). 복합이면 배열(예: [c,d], P21). use_example은 글쓰기 job_type **또는 gap_type에 c(재번역) 포함** 시 생성한다(재번역이 핵심인 비글쓰기 시나리오 누락 방지, P27).
- **[더 깊이] 후속 로직:** action="더깊이"면 프롬프트2가 해당 갈래의 하위·파생 선택지 3~4개 생성. 의중 분포는 갱신 않고 탐색 경로에만 기록(분기 방향만 바꿈).
- **단어 종류 + 상세 열람(P14):** 각 추천어에 `kind`(개념·방법론·지표·제도·도구·절차·역할)를 배지로 표시해 "이게 무엇인지" 한눈에. [자세히] 클릭 시 의미·세부 내용을 on-demand로 불러옴(프롬프트5, lazy — 5~8개를 미리 생성하지 않아 토큰·지연 절약).
- **[자세히] 드롭다운 정보 구조(P32, 핵심 학습 표면):** 펼치면 고정 섹션이 위에서부터 스트리밍으로 채워짐(한 번 연 내용 캐시, 좁은 패널이라 single-open). **섹션 2개** = ① **개념·내 맥락·활용**(무엇인가[비전공자 눈높이+비유] → 왜 내 상황에 중요[context_note] → 어떻게 쓰나·판단[use_example·direction·절차]을 한 흐름으로) ② **함께 볼 어휘 + 출처**(relates_to 칩[클릭 시 점프] + 참고 출처[필수]). 더 원하면 맨 아래 [더 깊이](예시·비유 추가, 기본은 짧게).
- **그룹핑 — 상위 주제 클러스터(P32):** 각 Term에 `group`(상위 주제 라벨)을 부여해 어휘를 의미 단위로 묶는다. 화면2 기본은 priority 평면 리스트(각 카드에 그룹 칩)이고 [그룹으로 보기] 토글로 클러스터별 재배치한다. group은 [지도 보기](§2)의 조직 원리로도 재사용한다. **역할 분리: group은 상위 부모, relates_to는 같이 봐야 할 동렬 어휘.**
- **어휘 더 보기 — 추가 생성(P32):** 추천은 5~8개로 시작하되 [어휘 더 보기]로 다음 우선순위 묶음(예: +5)을 프롬프트3 재호출로 추가 생성·스트리밍 append한다. 이미 노출·[알아] 태깅된 term은 중복 제외(exclude_terms로 전달, §5-1). on-demand 생성이라 비용↑ → freemium 가드(무료는 더보기 횟수 제한). 화면1 [이 갈래 더 깊이](좁히기 측 분기 확장)와 다른 별개 동작.
- **출처 명시 — 필수(P32, 저작권·신뢰):** 모든 개념 상세(프롬프트5 detail)에 참고 출처(RAG 검색 출처)를 표기한다. 원문 복붙이 아니라 LLM이 재서술한 설명 + 참고 출처(사이트명·제목·URL) 1~3개. 출처가 없으면(무RAG 강등) "확인된 출처 없음 · 모델 일반지식 기반"으로 명시하고 신뢰도 낮음을 알린다. 표기 형식·발췌 한도 정밀선은 §11 법무 확인 항목.

- **비용 제어:** 분야→어휘 지도 생성은 분야별로 캐시(같은 분야 반복 호출 방지). 선택지 생성은 세션 내 시스템 프롬프트 캐싱(DeepSeek-V4 지원 여부 확인 필요).

## 5. 프롬프트 설계 (LLM이 하는 일) — 종합안 + 시나리오 검증(§10) 반영

1. **진입 → 분야 + 작업유형 이중 추론 + 첫 분기** (MVP)
   - 입력: 분야명 또는 "하려는 말" 자유문장(진입점 B) + 붙여넣은 맥락(context_snippet, 선택).
   - 출력: 확정 분야 + **job_type(작업 유형: 보고서작성·의사결정·서류제출·이해학습·진단판단·문제해결·협상설득준비·전문가면담준비·기획전략)** + (선택)user_condition + 첫 질문 + 선택지 3~4개(Choice[]). 한 입력이 두 작업을 겸하면 job_type을 최대 2개까지(복수 job_type, P18).
   - 분야만이 아니라 작업유형을 함께 추론해 출발점을 넓힌다(P1·P2). 사용자가 이미 상황을 서술했으면 그 상황을 다시 묻지 말고 세부 의중을 가르는 선택지를 내라(fast-track, P5). 비전공·비전문 직군이 감지되면 선택지 언어를 그 역할 시선으로(P8).
   - 같은 분야 첫 1~2 분기는 캐시(공통 고정구간 §3-2).
2. **누적 선택 → 다음 분기** (MVP, 실시간, 지연 민감)
   - 입력: 분야 + job_type + 선택 이력 + 현재 후보 영역. 출력: 다음 질문 + 선택지 3~4개(Choice[]).
   - action="더깊이"면 직전 갈래의 하위·파생 선택지를 생성(의중 분포는 갱신 안 함, §3-3).
   - 매 선택 호출이라 지연·단가의 핵심. DeepSeek-V4 실측·지연 fallback(§7-3)이 중요.
   - **수렴 목표 = 3턴 (2026-06-25): enough·confidence로 3턴 안 마무리를 목표. 의중이 특정되면 enough=true, 정말 크게 갈릴 때만 추가(최대 8). 질문 채우려 늘리지 않음. 프론트 진행 바는 점이 늘지 않는 연속형(목표 3 기준, 점 팝 금지).**
3. **좁혀진 영역·상황 → 우선순위 추천 어휘** (MVP, 추천 엔진, P13)
   - 입력: 확정 영역 + job_type + user_condition + context_object + gap_type(프롬프트1·2 추론값, §3-3 P27).
   - **user_condition·context_object가 이 호출 시점(화면2)에 있어야 우선순위가 개인화된다.** 화면3에서만 받으면 추천이 일반론이 되므로 화면0~1에서 선택적으로 미리 수집한다(P16·P17, §3-2 fast-input).
   - **선정 원칙(P31 개정 2026-06-25): 사용자가 입력에 직접 쓴 용어(앵커)는 이미 아는 것으로 보고 terms에서 제외한다(되돌려주지 않음). 목적은 사용자가 AI에게 제대로 부탁하고 그 작업을 실제로 굴리는 데 필요한 '실용 어휘'(특히 방법론·기법·핵심 개념)를 쥐여주는 것. 앵커의 아래·옆·심화로 들어가 비전공자가 실전에서 막히는 must-know 5~8개를 고른다(표층 동의어·사용자가 이미 쓴 말의 재진술 금지). 정의는 짧고 쉽게(한 문장, 한자어·장황 금지).** 구버전("입력 명시 핵심어 반드시 포함")은 사용자가 이미 아는 앵커를 그대로 되돌려줘 제품 목적(배경지식 어휘력 확장)과 어긋나 뒤집음(사용자 directive). must-know 커버리지는 유지하되 그 대상이 앵커가 아니라 앵커를 둘러싼 실용 어휘다.
   - 출력: 이 상황 최우선 어휘 5~8개를 **우선순위 순으로** 추천(Term[], priority·why 포함). 단순 나열이 아니라 "당신 상황엔 이게 1순위"를 근거(why)와 함께 — 틀린 우선순위로 오도하지 않도록 근거를 반드시 동반.
   - 유형별 부가 — gap_type에 따라 출력 필드를 켠다(P7·P11·P12·P19·P27·P28·P30): a는 one_line(기본), b는 `relates_to`·`order`, c는 `use_example`+`direction`, d는 `direction`·`context_note`, e는 `context_note`+`direction`. **켜기 단위는 어휘별 — Term.gap_type가 있으면 그 값으로 대체(세션 기본값 무시), 없으면 세션 기본 gap_type을 쓴다(P30).** use_example은 글쓰기 job_type 또는 gap_type c 포함 시 생성하되, 둘 다면 재번역 예시를 우선한다(글쓰기 작성 예시는 보조). direction은 gap_type c·d·e 또는 job_type 의사결정·진단판단·협상설득준비에서 켠다(둘 중 하나라도 해당하면 켬, 합집합). 복수 job_type이면 각 job_type의 필드를 합집합으로 켠다.
4. **태깅 결과 → 정리/넘기기 텍스트** (MVP)
   - 입력: 영역 + job_type + [알아/몰라/적용모름] 태그 어휘 + user_condition + context_object + 배경 단서.
   - 출력 paste_text 고정 템플릿: **"나는 [분야] 영역에서 [task_intent]를 하려 한다. (내 상황: [user_condition]) 핵심어 A·B·C 중 B는 잘 모른다. (참고 맥락: [context_object])"** — 빈 슬롯은 생략. job_type이 복수면 task_intent에 두 작업을 함께 적는다(예: "내용을 이해하고 그걸로 이메일을 쓰려 한다", P18).

5. **단어 상세 (on-demand, P14)** — [자세히] 클릭 시에만 호출(lazy)
   - 입력: term + kind + area + job_type + rag_sources(검색 출처). 출력: 의미(정의) + 세부 내용 + 관련어 1~2 + (해당 시) 적용 예시 + **참고 출처(필수, P32 — RAG 출처 재서술, 원문 복붙 금지. 없으면 무출처 표기)**.
   - 5~8개를 미리 생성하지 않고 사용자가 연 카드만 호출해 토큰·지연 절약.

- **계측(deepflow C3):** 선택·태깅 클릭에 timestamp를 찍어 카드 렌더링부터 클릭까지 경과시간을 localStorage에 기록(§7-2 망설임 지표용).
- **출력 눈높이(P15):** 모든 LLM 출력(선택지·추천·why·상세)은 **비전공자 기본 가정**으로 쓴다. 배경 포착(§3-2)으로 경험자가 확인될 때만 상향 조정.

### 5-1. 인터페이스 계약 (구현 첫날 막힘 방지)
파이프라인이라 출력 형식을 못 박지 않으면 파싱이 무너진다. **전부 structured output(JSON) 강제.** DeepSeek-V4 JSON mode 지원 여부는 착수 전 확인(§1-1). 미지원 시 후처리 파서(코드블록·정규식).
- 프롬프트1 입력 = `{ raw_input, context_object? }`. 출력 = `{ domain, job_type, user_condition?, condition_required, question, choices: Choice[] }`. **condition_required(bool)는 조건 의존도 판정 결과 — true면 프론트가 화면0~1 조건 입력을 "권장"으로 승격(P29·P30, §3-2).** job_type 열거값 = 보고서작성·의사결정·서류제출·이해학습·진단판단·문제해결·협상설득준비·전문가면담준비·기획전략 (§3-2 작업유형 분기와 단일 출처, 키 변동 금지). 한 입력이 두 작업을 겸하면 job_type을 최대 2개 배열로 출력(복수 job_type, P18). 화면0 보조 입력에 조건을 적었으면 user_condition으로 함께 출력.
- 프롬프트2 입력 = `{ domain, job_type, history: [{label, action: "선택"|"더깊이"}], remaining_tags[], context_object?, user_condition? }`. 출력 = `{ question, choices: Choice[] }`. (action은 프론트가 사용자 클릭 종류에 따라 주입, 표기는 §3-3과 동일. context_object·user_condition이 있으면 그 맥락에 맞춰 선택지를 좁힌다 — 붙여넣은 문서·조건을 화면1부터 활용해 §3-1 이월 선언과 정합, P16·P17)
- 프롬프트3 입력 = `{ area, job_type, user_condition?, context_object?, gap_type?, exclude_terms?, more_count? }`. **gap_type은 프롬프트1·2가 추론한 session 기본값(닭-달걀 해소, P27)**, 열거값 = a·b·c·d·e, 복합이면 배열(예: [c,d], P21). **exclude_terms·more_count는 [어휘 더 보기] 재호출용(P32) — exclude_terms는 이미 노출·[알아]분, more_count는 추가 묶음 크기.** 출력 = `Term[]`(priority 오름차순 — 1이 최우선, §3-3). **Term[] 출력 필드: 필수 = term·kind·group·priority·why·one_line·tag; 조건부(gap_type/job_type에 따라 켬, P27·P28) = direction·use_example·context_note·relates_to·order·gap_type.** 유형 다음 필드 매핑(고정, P30): a는 one_line, b는 relates_to·order, c는 use_example과 direction, d는 direction·context_note, e는 context_note와 direction. direction은 gap_type c·d·e 또는 job_type 의사결정·진단판단·협상설득준비에서 켠다(합집합). use_example은 글쓰기 job_type 또는 gap_type c 포함 시, 둘 다면 재번역 예시 우선. 어휘별로 유형이 다르면 Term.gap_type로 대체 적용(세션 기본값과 합집합 아님). 복수 job_type이면 각 job_type의 필드를 합집합으로 켠다. **user_condition·context_object는 화면2 추천 시점에 이미 있어야 priority·why가 개인화된다(화면3 only면 일반론, P16·P17, §3-2 fast-input).**
- 프롬프트4 입력 = `{ area, job_type, vocab: [{term, tag}], user_condition?, context_object?, background_hint? }`. **user_condition은 화면3 우선 병합 최종값(화면0 값을 화면3가 덮음, P26).** 출력 = `{ area, task_intent, user_condition?, context_object?, context_sentence, vocab: [{term, tag}], paste_text }`. context_sentence는 프롬프트4가 background_hint에서 생성. job_type이 복수면 task_intent에 두 작업을 함께 담는다(P18). paste_text가 [복사]·[공유]가 넘기는 최종 한 덩어리(위 템플릿).
- 프롬프트5 입력 = `{ term, kind, area, job_type, rag_sources }`. 출력 = `{ definition, detail, related[], example?, sources: [{title, site, url}] }` (on-demand, [자세히] 시). **sources 필수(P32): RAG 검색 출처를 재서술해 표기(원문 복붙 금지), 없으면 빈 배열 + 프론트가 "확인된 출처 없음·일반지식" 표기.** **`related[]`(프롬프트5)와 Term의 `relates_to`(프롬프트3)는 다른 개념이다(P28): related = 상세 열람용 일반 관련어 텍스트, relates_to = 화면2에서 같이 묶어 봐야 하는 핵심 term 참조(관계 표현용). 이름·용도가 다르므로 프론트는 별도 컴포넌트로 렌더링.**
- Choice·Term 스키마(§3-3)를 단일 출처로 공유(키 이름 변동 금지).

## 6. MVP 범위 / 비범위

**범위(MVP):** 다진입로(상황칩·자유문장, **작업유형+분야 동시 포착**) → **실시간 적응형 다지선다 좁히기(B)** → **우선순위 추천 어휘 + 태깅(A)** → 정리(내 상황 추가 선택) → [복사]·[URL 공유]. 웹앱. 핵심 가치가설(클릭으로 의중 좁힘) 검증. 넘김 텍스트에 작업·조건·맥락 동반(§5), 세션 간 이월 포함.

**비범위(2단계 이후로 미룸):**
- 로그인·계정·서버 동기화 (localStorage 이월은 1차 포함)
- 브라우저 확장 + 페이지/대화 맥락 자동읽기(진입점 C). 단 진입점 B(자유문장)는 1차 부분 포함. 이미지 문서(사진·스캔) 자동 OCR도 1차 비범위 — 핵심 수치·항목 수동 입력으로 유도(P22, §2 화면0)
- 본격 개인화("나만의 과외선생", 원본 §5). 세션 간 이월은 1차 최소 버전만
- 메인 AI 자동 연동(클립보드 주입). 단 [URL 공유]는 1차 옵션
- 진짜 EIG 엔트로피 계산, 빈도가중 트리 (1차는 LLM-prompted)
- 크로스 도메인 어휘 연결 시각화 (1차는 domain_tags 복수 허용으로 기반만)
- 크로스브라우저, 다국어, 음성 입력

**MVP 통과 기준 (관찰 전에 못 박음, go/no-go):** 셋 다 충족해야 GO.
- (가치 획득, 가설 2·3) 5명 중 4명 이상이 "분야 어휘·개념을 쥐었다(이제 무엇을 물을지 안다)"에 동의하고 그 상태로 메인 AI 질문을 이어감 — 정리를 복사·공유하든 직접 입력하든 무관(복사율 단독은 GO 기준 아님).
- (재방문 의도, 가설 4) 3명 이상이 "다음 AI 대화 전 다시 열겠다"고 답하고, 그 이유가 메인 AI에 직접 묻는 것 대비 납득되는 차별.
- (메타인지, 가설 1·2) [알아/몰라] 망설임 3초 이상 비율이 과반 미만이고, 정리가 본인 의중과 맞다는 동의가 다수. (측정 §7-2)

이 기준은 §8 0단계(DeepSeek-V4 실측·경쟁 보강·프롬프트 품질 평가) 완료 이후 적용. 하나라도 미달이면 그 약점을 고쳐 재검증(무조건 진행 금지).

## 7. 검증 가설 · 관찰 프로토콜 · 리스크 (deepflow C1 재구성)

### 7-1. 검증 가설 (순서대로, 앞이 깨지면 뒤를 해석 못 함)
1. **메타인지 가설 (선결):** 사용자가 다지선다에서 자기 상황에 맞는 선택지를 고르고(재인), 어휘 태깅에서 [알아/몰라]를 *실제 지식과 일치하게* 누를 수 있는가. 깨지면 이후 클릭이 노이즈.
2. **의중 좁힘 가설:** 그 선택이 누적되어 정리가 사용자의 실제 의중을 맞히는가.
3. **전달 효용 가설:** 이 탭을 거친 뒤(정리를 넘기든 사용자가 직접 묻든) 메인 AI 대화 왕복이 줄거나 답이 나아지는가.
4. **존재이유 가설(§1-1):** 사용자가 메인 AI에 직접 묻는 대신 이 탭을 *먼저 열* 의지가 있는가. **대조 태스크(deepflow C3):** 같은 의도를 메인 AI에 바로 물었을 때와 이 탭을 거친 뒤 물었을 때의 첫 응답 적합도를 한 세션에서 비교(최소 5명 중 절반). 이게 방어 명제의 직접 검증.

### 7-2. MVP 관찰 프로토콜 (사용자 5명)
- 메타인지: [알아/몰라] 판단 망설임 시간과 3초 이상 비율, 뜻 보고 번복하는 오선택 비율.
- 좁힘: 정리를 보고 "내 의중이 맞다"고 동의하는지.
- 이해 전환: [자세히] 드롭다운 열람률과 [몰라]→[이제 알겠다] 전환율(개념을 실제로 쥐었는지의 1차 신호).
- 전달 마찰: [정리]까지 도달한 사람 중 **실제로 메인 AI에 붙여넣거나 공유해 완주한 비율**(이탈 지점 측정).
- 행동: 24시간 뒤 재방문 여부, 또는 최소한 "다시 쓸 것 같은가 + 이유".
- 분야 편차: 교과서적 분야(경제학)와 생소하거나 세부적인 분야에서 카드 품질이 갈리는지.
- 망설임 원인 분리: 세션 종료 후 1분 인터뷰 "가장 고민된 순간이 언제, 왜였나"로 버튼 의미 오해(UI 문제)와 지식 불확실성(카드 문제)을 사후 분리.
- **피험자 모집(deepflow C3):** ① 해당 분야 비전문가(학부 수준 이하), ② 메인 AI를 주 1회 이상 쓰는 자. 비전문가를 의도적으로 과표집(메타인지 실패 조건 탐지).
- **공통 고정구간(deepflow C3):** 같은 분야 첫 1~2 분기 선택지는 5명 모두 동일 제시(§3-2)해 메타인지(공통구간)와 좁힘(적응구간)을 분리 해석.
- **측정 모달리티(deepflow C3):** 망설임은 클릭 timestamp 코드 계측(§5 계측)으로 객관화. 관찰자 눈대중 금지.
- **세션 스크립트(deepflow C3):** 도입("이걸 쓰기 전 어떤 상황이었나"), 탐색 중 진행자 침묵, 종료 인터뷰 3문("가장 고민된 순간과 왜", "메인 AI에 바로 묻고 싶던 때가 있었나", "다시 열 것 같은가와 이유"). 진행자가 달라도 동일 조건.
- **작업·맥락 전달 점검(§10):** 정리 텍스트(paste_text)에 작업 의도·개인 조건·문서 맥락이 실제로 담겨 AI 답이 일반론을 벗어나는지 관찰(P1·P3·P4). 자유문장 진입 비율과 수동 입력 마찰 이탈도 측정(진입점 C 우선순위 근거).
- **조건 fast-input 마찰 vs 개인화(P16·P17):** 화면0~1 보조 조건 입력의 입력률·이탈률과, 조건을 넣은 세션의 추천 개인화 체감(우선순위가 내 상황에 맞았는가)을 함께 측정해 "선택→권장" 승격의 득실을 본다.

## 8. 개발 로드맵

1. **0단계 (착수 전):** DeepSeek-V4 실측(단가·지연·한국어·캐싱·JSON mode) + 한국 로컬 경쟁 보강 조사 + 메인 AI 3종에서 "어휘 카드와 클릭 좁히기" 흐름 수동 재현 시도(차별점, §7-3) + **프롬프트1·2·3 품질 평가**(시나리오 대표 5~6개 고난도 분야 수동, use_example·direction·relates_to 정확도 포함, §3-2 품질 가드 P24) + **추천 "상황 문장" 칩 10~15개 확정**(분야명 아닌 상황 문장으로, vocab_gap 크고 비전문가 진입 잦은 것 우선, P9) + **메인 AI deep link 가능 여부 점검**(§2 화면3).
   - **[2026-06-22 0단계 일부 선이행]** thinking·RAG 격자 실측 완료(`0단계-실측-결과.md`): 채택 레시피 = **pro + thinking OFF + RAG + term 단위 스트리밍**(유일하게 정확도 85 통과, 85.4). thinking ON은 TTFT 17~24초로 탈락(OFF는 0.8초), RAG는 최신성 오류 100% 교정·정확도 +25~27점. **남은 0단계:** 단가·캐싱·대규모 부하 측정, RAG 소스 신선도 운영, 도메인 위험도 분류(저위험 flash+RAG / 고위험 pro+RAG 하이브리드 라우팅), term 스트리밍 출력 포맷 구현 + **하드 도메인 pro 폴백**(빌더 검증서 flash+RAG가 경계선인 도메인=재무모델·결제정산·AR/VR 등은 pro로 라우팅) + **구현기 도메인별 프롬프트3/RAG 튜닝**(전역 튜닝만으론 전 도메인 동시 85 불가 확인, P31).
2. **1단계 (MVP):** §6 범위. §7-1 가설(메타인지, 좁힘, 전달, 존재이유)을 순서대로 확인하고 **§6 통과 기준 충족 시에만** 2단계로.
   - 마일스톤(동작 기준): ① 진입·첫 분기·선택 기록, ② 실시간 적응형 다지선다 동작(지연 허용선 = §7-3 fallback 기준 한국 TTFT 1.5초 이내, 초과 시 fallback), ③ 어휘 태깅·정리·복사/공유, ④ 사용자 관찰. 앞 단계가 동작해야 다음으로.
3. **2단계:** 진짜 EIG 엔트로피 계산, 크로스 도메인 어휘 연결 시각화, 분야 다양화, 정리 품질 개선.
