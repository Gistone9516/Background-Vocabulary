# 배경어휘 사이드탭 — 프론트↔백엔드 인터페이스 계약 (SoT v1.0, 2026-06-25)

> buildflow ②단계 SoT. **충돌 시 이 문서가 권위(SoT wins).** 폴더별 spec은 여기에 conform.
> 상위 정본: 제품 동작·UX·필드 = `panel.html`(최신 기획). 인프라 제약(로케일·RAG·안전 게이트) = 백엔드.
> 코드 현재값 출처: `sidetab/packages/shared/{types,interfaces,pipeline.stub}.ts`, `prompts/index.ts`. 본 문서는 그 위의 **델타**.

## 0. 권위 규칙
- 화면 흐름·노출 필드·문구·태깅 UX = **panel.html이 권위**. 백엔드가 굽힌다.
- 로케일 라우팅·RAG 필수·고위험 거부·TTL·모델 라우팅 = **백엔드가 권위**. 프론트가 따른다.
- 프론트는 전부 동기 mock(SEED/MORE, setTimeout). 실제는 5엔드포인트 + SSE 스트리밍으로 분리.

## 0-1. ★ 이식성 불변 경계 (AWS Lambda 이식 대비 — 1순위 제약, 구현계획 §3-1)
> 이 프로젝트는 Cloudflare Workers 지금 / AWS Lambda 나중을 **어댑터 추가만으로** 가능해야 한다. 본 SoT의 어떤 변경도 이 경계를 깨면 안 된다. 빌드 에이전트는 이 절을 먼저 읽고 따른다.
- **core/·shared/ 안 런타임 전역 0건.** Workers 전용 바인딩(KV·Durable Objects·`caches.default`·`ctx.waitUntil`)·Node 전용 API 직접호출 금지. 외부 자원은 전부 `LlmClient·SearchProvider·CacheStore·EnvConfig` 인터페이스 뒤로 추상화하고 주입(DI).
- **웹표준만 사용:** fetch · Request/Response/Headers · Web Streams(ReadableStream·TransformStream) · Web Crypto · TextEncoder/Decoder · URL · AbortController/AbortSignal. (전부 Node 18+/Lambda 동작.)
- **런타임·공급자 특수성은 `adapters/`에만:** Workers env 매핑·Hono 부트·StreamEvent↔SSE 직렬화·tier 게이팅(헤더 검사)·Upstash/Tavily REST 호출.
- **Lambda 구현체는 지금 안 짠다.** 인터페이스 경계만 유지. `adapters/lambda/`는 트리거 시 추가(Hono 흡수, 50~100줄). 인터페이스를 Lambda 시그니처에 미리 넓히지 않는다.
- **이번 변경 점검 결과 = 런타임 누수 0:** enough/confidence·domain_risk·exclude·Source·Prompt5 3단 = 순수 데이터. signal = AbortSignal(웹표준). /summarize 게이팅 = 어댑터 라우트. site = URL 호스트 파생(웹표준).
- **하드플로어 가드(사이클1 신규):** `core/`·`shared/`가 런타임 전용 모듈 import 또는 런타임 전역 참조 시 빌드 실패하는 경계 테스트 추가.

## 2. 공유 타입 변경 (shared/ 델타)

### 2-1. 신규 타입 `Source`
```ts
export interface Source { title: string; site: string; url: string }
// site = URL 호스트에서 파생(어댑터/core 책임). RagDoc에는 site 없음 → 변환.
```

### 2-2. `Term` (P3 출력) — 필드 추가
```ts
// 추가: group?  (그룹뷰 toggle용, 프론트 t.group)
group?: string;
// 기존 relates_to 유지(프론트 'relates'는 relates_to로 통일 — 프론트가 백엔드명 채택)
```
- `kind`는 자유 문자열 유지(프론트 KIND_HUE 7종은 표시 힌트일 뿐 제약 아님).
- `tag`는 항상 "몰라"로 생성(기존 유지). 미태깅 상태는 프론트 전용(§4-1).

### 2-3. `Prompt2Out` — 종료 신호 추가 (D1)
```ts
export interface Prompt2Out {
  question: string;
  choices: Choice[];
  enough: boolean;     // 신규: true면 더 묻지 말고 recommend로(백엔드 판단)
  confidence: number;  // 신규: 0..1, UI 힌트("거의 다 좁혔어요")
}
```
- **엔벨로프 규칙(프론트 가드):** 최소 3질문 전에는 `enough=true`여도 계속, 8질문 도달 시 강제 종료. 즉 백엔드가 [3,8] 안에서 결정.
- 진행 UI: 고정 분모 제거. "AI가 좁히는 중 · N번째" + confidence 높으면 힌트. 점은 누적형.

### 2-4. `Prompt5In` / `Prompt5Out` — 구조 재정의 (D2)
```ts
export interface Prompt5In {
  term: string; kind: string; area: string; job_type: JobType[];
  domain: string; topic: string; locale: Locale;  // 신규: recommend RAG 캐시 재사용 키
  deepen?: boolean;                                 // 신규: "더 깊이(예시·비유 추가)"
}
export interface Prompt5Out {           // 프론트 상세 3단 구조로 재정의
  what: string;        // 개념 (프론트 t.what)
  whymine: string;     // 내 맥락 (t.whymine)
  how: string;         // 활용 (t.how)
  misc?: string;       // 보조 콜아웃 (t.misc)
  related: string[];   // 함께 볼 어휘 (일반 관련어, relates_to와 별개)
  sources: Source[];   // 출처 필수. 빈 배열이면 프론트 "확인된 출처 없음" 폴백
  example?: string;    // deepen 시 추가 예시
}
// 폐기: definition, detail (what/whymine/how로 대체)
```

### 2-5. `RecommendInput` — 페이지네이션 + 안전 + 취소

## 5. 화면별 계약 상세
- **entry:** `S.input`→raw_input. `S.cond`(조건 추가)→user_condition. context_object는 명시적 붙여넣기 시만(현재 별도 필드 없음 → MVP 빈값). 고위험 정규식 1차 통과해도 /classify domain_risk가 권위.
- **narrow:** /classify 첫 질문 → /next 루프. 정적 stepDefs 트리는 mock이라 폐기, 질문은 전부 백엔드 생성. 다중선택 2개·되돌리기 → Prompt2In.history `{label, action:'선택'}`. "지금 충분해요" = 수동 조기 종료(enough 무시하고 recommend).
- **terms:** /recommend SSE, term 도착마다 stagger(`_new`). "더 보기" → exclude=현재 term명들로 2차 /recommend. groupView=t.group. 태깅은 클라 상태 + chrome.storage.local('known' 영속).
- **summary:** 기본 = 클라 buildSummary(즉시). "AI로 더 정리"(유료) = /summarize에 vocab+tag+condition 실어 호출.
- **paywall:** free 주7회(FREE_WEEKLY_LIMIT). 429 시 limitHit UI. remaining 표시는 클라 추정(게이팅 Tier3 전까지 cosmetic).
- **refusal:** classify domain_risk=high 또는 프론트 정규식. **부드러운 거부** — 차단만 하지 않고 왜 못 다루는지 이유를 설명하고 재시도(다른 맥락으로) 유도. 사용자向 문구 정본 = panel.html `screenRefusal`("이 분야는 도와드리기 어려워요 / 의료·법률처럼 개인의 진단·판단이 필요한 고위험 영역은 다루지 않아요. 기술·창작·비즈니스 학습 맥락에서 다시 시도해 주세요"). 백엔드 HIGH_RISK_REFUSED는 신호일 뿐, 표시 문구는 프론트 화면이 권위.

## 7. 열린 항목 (구현 중 점검)
- **D2 출처 근거 정확도(정밀 우선, 레드펜 3=ⓐ):** detail이 recommend RAG 캐시(rag:domain:topic:locale) 재사용해 term별 관련 출처 추출. **확신 가는 귀속만 포함, 애매하면 `sources:[]`(빈 배열)로 두고 프론트 "확인된 출처 없음" 폴백** — 거짓 귀속 금지(틀린 출처가 신뢰를 더 깎음). 매핑 정확도 실측, 부족하면 term별 경량 재검색. `site`는 URL 호스트 파생(RagDoc엔 없음).
- **exclude 페이지네이션:** P3 프롬프트가 "이미 제시한 어휘 제외, 다음 우선순위"를 일관 준수하는지 검증(중복·고갈 처리).
- **enough 캘리브레이션:** [3,8] 엔벨로프에서 confidence 임계 실측(너무 일찍/늦게 종료 방지).
- **remaining 정확도:** 익명 게이팅(설치 UUID, Tier3) 전까지 클라 추정.

## 8. Phase 0 — 결제·인증·엔타이틀먼트 계약 (SoT, 2026-06-29)

> buildflow ②단계 Phase 0 계약. **이 절이 결제·인증 구현의 SoT.** 상위 설계 = `배경어휘-사이드탭-결제인증-설계.md`(§A·§K·§L). 충돌 시 본 절(machine-verifiable)이 권위.
> **범위 = 기반 5종(외부 계정 불요).** PG 어댑터 실구현·webhook·dunning·Cron·이메일은 Phase 1+.

### 8-0. 확정 결정 (배경: 결제인증-설계)
- 인증 = Google SSO `launchWebAuthFlow` + 내부 `user_id` + 자체 HS256 JWT. `chrome.identity getAuthToken` **폐기**(§L). `x-tier` 헤더 신뢰 **완전 제거**(§A4).
- 엔타이틀먼트 SoT = Cloudflare **D1**. 포트(`UserRepository`)=shared, 구현(`D1UserRepository`)=adapters. **core/shared의 D1 참조 0**(이식성 0-1 불변).
- 가격 국내 ₩3,900 / 해외 $2.99(세금 별도). PG = 포트원(Phase 1).
- **★ §A8 정정(b안 확정, 2026-06-29):** `PAYMENT_ENABLED=false`면 **"전원 pro" 폐기** → 티어는 평소대로(JWT pro면 paid, 없으면 free). 결제 라우트·페이월 CTA만 "준비 중". 근거 = freemium value gap 유지(웹근거). 개발 중 pro 테스트 = `DEV_MODE` 가드부 `DEV_FORCE_TIER`(프로덕션 무영향).
- **시각 단위 경계:** JWT `iat`/`exp` = epoch **초**(JWT 표준). D1·도메인 시각(`expires_at` 등) = epoch **밀리초**. 변환은 adapters에서.

### 8-1. 공유 타입 (shared/types.ts 추가)
```ts
export const SUBSCRIPTION_STATUSES = ["none","active","canceling","past_due","grace","expired","suspended","refunded"] as const;
export type SubscriptionStatus = (typeof SUBSCRIPTION_STATUSES)[number];
export type PgProvider = string;          // 리터럴 union 금지(어댑터 추가 시 SoT 무수정). 검증은 어댑터.
export type Currency = "KRW" | "USD";

// D1 users 행의 도메인 표현. PK=내부 user_id, email로 소유(google_sub 단독 PK 금지, §L3).
export interface UserRecord {
  user_id: string; email: string; google_sub: string | null;
  tier: Tier; subscription_status: SubscriptionStatus;
  expires_at: number | null; current_period_end: number | null;
  cancel_at_period_end: boolean; grace_until: number | null;
  failed_payment_count: number; next_retry_at: number | null; last_failure_code: string | null;
  current_price: number | null; currency: Currency | null;
  billing_interval: "monthly" | "yearly"; pg_provider: PgProvider | null; created_at: number;
}
// 서버가 판정한 유효 권한(JWT에 실리는 부분집합). grace 중 pro 유지.
export interface Entitlement {
  user_id: string; effective_tier: Tier;   // (tier=paid) OR (grace_until>now) ? paid : free
  subscription_status: SubscriptionStatus; expires_at: number | null;
}
// 자체 액세스 토큰 클레임(HS256, §A4·K4). 시각=epoch 초.
export interface AccessTokenClaims {
  sub: string;            // 내부 user_id
  tier: Tier;             // = effective_tier(서버 판정)
  email: string;
  iat: number; exp: number;   // 발급+15분
  jti: string;            // 추적용
}
// 리프레시 토큰 클레임(30일, stateless). /auth/refresh에서만 사용. 시각=epoch 초.
export interface RefreshTokenClaims {
  sub: string;            // 내부 user_id
  typ: "refresh";         // access와 구분(access엔 typ 없음)
  iat: number; exp: number;   // 발급+30일
  jti: string;            // 블랙리스트 키(K4, refresh 시점 검사)
}
```

### 8-2. 포트 (shared/interfaces.ts 추가) — 구현은 adapters
```ts
export interface NewUser { email: string; google_sub: string | null }
export interface EntitlementPatch {        // webhook/Cron이 적용. occurred_at으로 순서 역전 가드.
  user_id: string; occurred_at: number;
  tier?: Tier; subscription_status?: SubscriptionStatus;
  expires_at?: number | null; current_period_end?: number | null;
  cancel_at_period_end?: boolean; grace_until?: number | null;
  failed_payment_count?: number; next_retry_at?: number | null; last_failure_code?: string | null;
  current_price?: number | null; currency?: Currency | null; pg_provider?: PgProvider | null;
}
export interface UserRepository {          // 엔타이틀먼트 저장소 포트(§A3). D1↔DynamoDB 교체 가능.
  findById(userId: string): Promise<UserRecord | null>;
  findByEmail(email: string): Promise<UserRecord | null>;
  findByGoogleSub(sub: string): Promise<UserRecord | null>;
  create(rec: NewUser): Promise<UserRecord>;
```

```
  BillingKey.Issued         → 빌링키 저장(가입)
  미인식                     → payment_events 기록만(방어적)
★blacklist(R1 MISSING — 환불 후 15분 pro 누수 차단): 취소·환불·분쟁 시 Upstash `blacklist:{user_id}` EX(남은 access exp). resolveTier(8-6)는 networkless 유지하되 이 이벤트엔 즉시 짧은-exp JWT 강제 재발급(/subscription/status가 반환).
```

### 9-6. 구독 상태 FSM (K3)
`none → active → (past_due → grace) → canceling → expired`, + `suspended`(분쟁)·`refunded`.
- **단일 원자 갱신(R1):** tier·subscription_status·expires_at·grace_until·last_event_at를 **하나의 SQL UPDATE 또는 db.batch**로(복수 독립 prepare().run() 금지). `applyEntitlement` 포트 주석에 제약 명기.
- **graceful cancel(R1 missing_axis):** 해지 요청 = `cancel_at_period_end=true`(즉시 free 아님). current_period_end까지 pro 유지 → 경량 Cron(9-7)이 기간 종료 시 tier=free + `cancelSchedules`+`deleteBillingKey`.
- 유효 pro = `effectiveEntitlement`(tier=paid OR grace_until>now).

### 9-7. 정기결제·dunning (1b, K2) — 포트원 예약결제 구동 + 경량 Cron (R1 채택)
- **갱신(자체 월청구 Cron 폐기):** 가입·매 결제 성공 시 다음 달 timeToPay로 `schedulePayment` 등록 → 포트원이 청구·`Transaction.Paid` webhook → 9-5가 period 연장 + 다음 달 재예약. **"오늘 청구할 사람 찾기" 쿼리 자체가 없어 DATE()/UTC·KST/billing_locks/CPU 버그를 통째로 회피.**
- **경량 일일 Cron**(`wrangler.toml [triggers] crons`, `scheduled` 핸들러 export 필요): ① grace 만료자 강등 ② `cancel_at_period_end` 기간 종료자 강등 ③ reconciliation(9-8). 시각 비교는 **정수**(`current_period_end <= unixepoch()*1000` — R1: epoch ms에 SQLite `DATE()` 쓰면 NULL → 전원 미갱신 조용한 실패). 1회 처리량 상한(예 ≤50건)+잔여 다음 주기.
- **grace:** Failed webhook 시 past_due·failed_count++. pgMessage가 카드만료/재등록류면 즉시 grace + 카드재등록 이메일. 일시실패는 다음 `schedulePayment`를 D+1/D+3/D+6로 재예약([확인필요: 포트원 예약 자동재시도 — 미지원이면 우리가 재예약]). grace 3~7일 만료+소진 → 강등. effective_tier는 grace 중 pro 유지.
- **★강등 순서(R1):** `cancelSchedules(billingKey)` → `deleteBillingKey` → tier=free. 첫 단계 실패 시 status=degradation_pending + 운영 알림(예약 남은 빌링키 삭제는 에러).

### 9-8. reconciliation (1b, K3)
경량 일일 Cron(9-7)에 통합하거나 ≥1h 간격(R1: <1h scheduled 핸들러는 CPU 30초 한도 → 전수조회 시 조용한 타임아웃). **전수 스캔 금지 → 필터:** 최근 24h 예약 청구건 중 webhook 미수신(payment_events에 해당 paymentId Paid 없음)·`REQUESTED` stuck 건만 `getPayment` 단건조회로 동기화. 1회 ≤20건, 잔여 다음 주기. 멱등.
