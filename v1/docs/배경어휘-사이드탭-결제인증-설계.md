# 배경노트 — 결제·인증·엔타이틀먼트 설계 (v1, 웹근거)

> 목표 = pro(구독) 결제·인증·엔타이틀먼트를 **전부 구현**하되 출시 시 결제는 피처플래그로 **비활성** 가능. 미구현 채로는 출시하지 않는다.
> 방법 = 백그라운드 Workflow로 결제사·인증·영속 6주제 웹조사(출처 포함) + 아키텍처 6렌즈(비용·보안·이식성·운영·UX·법규) 적대 평가 2라운드. opus 종합.
> 근거 강도 표기: [H]=다출처 확인, [M]=단일/추정 혼재, [확인필요]=미확정. 결정을 가르는 사실은 본문에 출처 URL.

## 0. 한 줄 결론

**PG부터 고르지 말고 "포트(인터페이스) + 선행 인프라"부터 구현한다.** 6렌즈가 R2에서 만장일치로 수렴한 것은 PG 선택이 아니라 그 **앞에 와야 할 4가지**다: ① `PaymentGatewayPort` 추상화 ② Cloudflare D1 엔타이틀먼트 SoT ③ 서버 발급 서명 JWT(현재 `x-tier` 헤더 신뢰 구멍 제거) ④ 한국법 유료전환 고지 체계. PG(토스·포트원·Paddle)는 이 포트 뒤에 어댑터로 꽂으며, 어느 것을 켜느냐는 **가격·타깃 결정**에 종속된다(§B).

> **결정 확정(2026-06-29, 사용자):** PG = **포트원 단일 SDK**(국내 토스페이먼츠 빌링 + 해외 페이레터/엑심베이 분기, 사용자 원안 "이중등록"). 가격 = **국내 월 3,900원 / 해외 월 $2.99(세금 제외)**. Paddle(소액 실효 ~25% 적자)·Stripe(한국 법인 불가, 2026-06)는 보류. 따라서 §B는 포트원 경로로 확정, 착수 대상은 Phase 0(§A·§D).

## 1. 맥락과 현재 코드의 구멍 (실측)

- 스택: 크롬 MV3 사이드패널 확장 + Cloudflare Workers(stateless) + Upstash Redis. 1인 개발. pro 월 3,500원(원화).
- 현재 pro는 **localStorage 토글**일 뿐 실제 결제 없음(`onUpgrade` = `localStorage.setItem("sidetab:plan","pro")`).
- **렌즈가 실제 코드에서 확인한 구조적 취약점:**
  - `workers/src/index.ts`의 `readTier()`가 요청 헤더 `x-tier`가 `paid`면 그대로 신뢰(서버에 엔타이틀먼트 SoT 없음). DevTools에서 헤더만 바꾸면 pro 기능 즉시 개방.
  - `/summarize`·`/detail`은 tier만 보고 그 tier 출처를 보증하지 않음. `x-user-id`도 자기신고라 매 요청 새 UUID로 보내면 주간 한도 무한 우회(코드 주석이 "Tier3 전까지 구조적 잔존"으로 자인).
  - `detailLimitFree`(무료 상세 3회)는 클라 state로만 집계 → 새로고침·직접호출로 무력화.
  - 전역 일일 캡·레이트리밋이 `fail-open`(Upstash 장애 시 통과)이라 유일 backstop이 단일 외부 의존.
- 결론: **결제를 "켜는 순간" 이 구멍들이 그대로 드러난다.** 그래서 결제 인프라보다 서버 엔타이틀먼트 검증이 먼저다(보안·이식성·운영·UX·법규 렌즈 R2 만장일치).

## 3. 인증·영속 웹조사 요약 (출처)

- **인증 = `chrome.identity` getAuthToken(Google OAuth), Google `sub`를 사용자 PK.** 마찰 0, 토큰 캐싱 자동(MV3 서비스워커 지원). UUID 디바이스 바인딩은 결제 연동 시 재설치·교차기기·환불 대응 불가 → **보조(미인증 세션 추적)로만**. (Brave 등 비크롬 지원 필요 시 `launchWebAuthFlow` 대안.) https://developer.chrome.com/docs/extensions/reference/api/identity
- **엔타이틀먼트 SoT = Cloudflare D1(SQLite).** Upstash/KV는 쓰기 전파 ~60초 eventual consistency → webhook 직후 tier 판별 오염. D1은 SQL 트랜잭션 원자 upsert + 프리 150M reads·3M writes·5GB(저트래픽 비용 ~0). https://developers.cloudflare.com/d1/platform/pricing/ · 저장소 비교 https://developers.cloudflare.com/workers/platform/storage-options/
- **Upstash = 역할 축소(세션 캐시 + rate-limit + RAG 캐시 전용).** SoT 아님.
- **webhook → D1 atomic upsert(멱등).** Stripe `entitlements.active_entitlement_summary.updated` 또는 PG별 이벤트, `Stripe-Signature` HMAC 검증(`constructEventAsync`, Workers `node_compat=true`), `wrangler secret`로 시크릿 주입. https://docs.stripe.com/billing/entitlements · https://hono.dev/examples/stripe-webhook
- **서버 발급 서명 JWT로 `x-tier` 제거.** 클라가 Google ID 토큰 제출 → Workers가 Google tokeninfo로 **최초 1회** 검증 → D1 tier 조회 → `{sub,tier,exp}` JWT(HS256, exp 1h) 발급 → 이후 모든 요청은 **JWT 서명만 검증**(per-request 외부호출·D1 재조회 없음, UX 렌즈 지적 반영). 취소 즉시 반영은 짧은 exp + 갱신 엔드포인트(+ 필요시 Upstash 블랙리스트).
- CWS: 자체결제 폐지(2021), **외부결제 허용**. 확장 내 결제폼 금지 → `chrome.tabs.create`로 외부 hosted checkout. https://developer.chrome.com/docs/webstore/program-policies/accepting-payment/

## A. 확정 사항 (PG와 독립, "전부 구현"의 뼈대 — 렌즈 만장일치)

이 8개는 어떤 PG를 켜든 동일하게 필요하다. **결제 비활성 출시여도 이 인프라는 동작**해야 "구현 완료"다.

- **A1 `PaymentGatewayPort` 먼저.** `adapters/workers/src/payment/port.ts`에 `createCheckout`·`verifyWebhook`·`cancel`·`refund` 인터페이스. 토스/포트원/Paddle은 각각 어댑터 1파일. PG 전환=포트 교체. 이식성 렌즈 결론: "스택을 먼저 고르지 말고 포트를 먼저 정의하라."
- **A2 인증.** ~~`chrome.identity` Google OAuth, `google_sub` = 사용자 PK~~ → **§L에서 재결정: getAuthToken 폐기, 내부 계정(email) + 자체 JWT, front-door = 매직링크(권장)/소셜/Clerk.**
- **A3 엔타이틀먼트 SoT = D1 + `UserRepository` 추상화.** `users{ google_sub PK, tier, subscription_status, expires_at, pg_provider }`(컬럼을 처음부터 둬서 후일 PG 추가 시 마이그레이션 0). `adapters/workers/src/db/d1-user-repo.ts`에만 D1 바인딩. **core/shared D1 참조 0 확인됨**(이식성 불변 유지, Lambda 이전 시 `DynamoDBUserRepository`로 교체).
- **A4 서버 서명 JWT.** `readTier()`를 JWT claims에서 읽도록 교체. 유료 엔드포인트(`/summarize`·`/detail`·`/recommend`·`/relate`·`/next` 등) JWT 검증.
- **A5 webhook 수신.** HMAC 서명 검증 → D1 atomic upsert, 멱등(중복 수신 대비). 시크릿은 `wrangler secret`.
- **A6 서버측 미터링(Tier3 B1·B3 봉합).** 주간 한도·detail 횟수를 D1/Upstash 서버 카운터로(클라 신뢰 제거). `google_sub` 기준이라 헤더 교체 우회 불가.
- **A7 한국법 유료전환 고지 체계.** 무료→유료 전환·가격 인상 시 **30일 전 고지 + 7일 전 리마인드**(전자상거래법·소비자기본법). 이메일 발송 + 동의 기록 DB. **PG와 무관하게 필수** — 없으면 피처플래그를 켤 수 없다. https://docs.stripe.com/billing/subscriptions/kr-card
- **A8 피처플래그 `PAYMENT_ENABLED`.** Workers env. **★b안 확정(2026-06-29): "전원 pro" 폐기** — false면 티어 게이팅은 평소대로(JWT가 pro면 paid, 없으면 free)이고 checkout 라우트와 페이월 CTA만 "준비 중" 스텁. 근거 = freemium value gap 유지(모두에게 pro 무료 지급은 업그레이드 유인 소멸과 회수 시 takeaway 반감, 웹근거). 개발 중 pro 테스트는 `DEV_MODE` 가드부 `DEV_FORCE_TIER`(프로덕션 무영향). true 전환의 **하드플로어 = A1~A7 완료 + 고지 30일 경과**.

## B. PG 선택 (사용자 결정 — 가치/우선순위 분쟁, fact 분쟁 아님)

> **확정: 포트원 단일 SDK(국내 토스 빌링 + 해외 페이레터/엑심베이), 가격 국내 월 3,900원 / 해외 월 $2.99(세금 제외).** 아래는 결정 배경과 보존할 소수의견.

렌즈는 여기서 갈렸고, **이건 fact가 아니라 사용자의 가치 선택**이다(=납작하게 만들지 않고 사용자가 결정).

- **축1 가격:** 3,500원 유지 → 토스/포트원만 흑자(Paddle은 ~25%로 적자). 5,000원+·연간 플랜 → Paddle도 ~10%대 가능.
- **축2 타깃:** 국내 우선 → 토스 단독으로 출발. 즉시 글로벌(다국어 ko/en/ja/zh 매출 천장 회피) → 해외 결제수단 필요.

**사용자의 명시 의도("국내 한국 PG + 해외 글로벌 PG 이중등록")에 가장 정합하는 경로:**
- **포트원 단일 SDK** = 국내(토스페이먼츠 빌링) + 해외(페이레터 글로벌/엑심베이) 분기를 **한 연동으로**. 월 5천만원 이하 플랫폼료 0원이라 초기 비용은 PG 수수료만(국내 토스 ~4.3% + 포트원 0.5%). 이중등록이 PortOne 하나로 성립(웹조사 확인). [M — 해외PG 수수료·소액 최소금액은 상담/확인필요]

**렌즈 권장(다수): 토스/포트원 우선, Paddle은 조건부.**
- 비용·보안·이식성·UX 렌즈: 3,500원 유지면 토스(포트원 경유)로 출발, Stripe는 한국법인 열릴 때까지 보류, 글로벌은 포트원 어댑터 추가.

**소수의견(보존 — ops·legal 렌즈, R2에서도 유지):** **Paddle 단독 MoR.** 근거 = 세무(VAT/판매세) 전세계 대행 + 법인 불요 + 구독 수명주기(갱신·취소·재시도) Paddle 관리라 1인이 스케줄러를 직접 안 짬 → 6개월 운영 붕괴 리스크 최소. **단 전제: 단가를 5,000원+/연간으로 올려 수수료 실효 20%↓.** 반론(다수): MoR도 A1~A7(특히 D1+JWT·고지)은 그대로 필요해 "운영 간소화" 실질 절감폭은 과장. 그리고 Paddle은 영국법인이라 국내 분쟁조정 관할 밖(집행 불확실).

## C. 구독 수명주기 (포트 뒤 어댑터가 흡수)

- 신규: checkout 성공 webhook → D1 `subscription_status=active, expires_at`.
- 갱신/실패: 토스 직결이면 **스케줄러 자체 구현**(Cloudflare Cron Trigger 또는 Upstash QStash) + 카드 만료 재시도 + grace. Paddle/포트원-해외PG는 플랫폼이 일부 관리. (이 부담 차이가 §B 소수의견의 핵심.)
- 취소/환불: 빌링키 삭제 또는 PG cancel/refund API. 청약철회 14일(국내) 고려.

## D. 단계별 구현 계획

- **Phase 0 (PG 무관 선행 — 지금 착수):** A1 포트 → A2 인증 → A3 D1+UserRepository → A4 서버 JWT(`readTier` 교체) → A5 webhook → A6 서버 미터링 → A7 고지 체계 → A8 플래그. 결제 OFF로 main 머지 가능(전원 pro). **리팩토링 브랜치와 충돌 피하려 결제 라우트는 OFF로만 진입.**
- **Phase 1 (어댑터):** 결정된 PG 어댑터 1개 + (병행) 사업자등록·통신판매업·토스 빌링 심사 신청. Paddle 갈 거면 Paddle 어댑터.
- **Phase 2 (활성화):** A1~A7 완료 + 고지 30일 경과 확인 → `PAYMENT_ENABLED=true`.

## K. 시나리오 검증 보강안 (2026-06-29 — 8클러스터 워크스루 + 적대검증)

> 사용자 결제 시나리오 8클러스터(신규구독·정기갱신·취소/환불·복구/멀티디바이스·엔타이틀먼트 정합·지역/세금·법규/고지·업다운/재구독)를 설계에 단계별 워크스루(run wf_2ac2a87e-d8b, 16에이전트)해 확정된 갭과 보강. real/partial 다수, **blocker 7테마**. 이 절이 Phase 0 구현의 실제 스펙이며 위 §A·§C·§H·§I를 구체화한다.

### K1. D1 스키마 최종 (보강 반영)
- **users**: `google_sub` PK · **`email`**(고지·복구 필수, OAuth email 스코프로 수집) · `tier` · **`subscription_status`** ENUM(active·canceling·past_due·grace·expired·suspended·refunded) · `expires_at` · **`current_period_end`** · **`cancel_at_period_end`** BOOL · **`grace_until`** · **`failed_payment_count`** · **`next_retry_at`** · **`last_failure_code`** · **`current_price`** · **`billing_interval`**(DEFAULT 'monthly', 연 플랜 대비 선행) · `pg_provider` · `created_at`.
- **payment_events**: `event_id` PK · **`cycle_key`** UNIQUE(google_sub + billing_cycle, Cron 이중청구 방지) · **`event_type`**(paid·failed·canceled·refunded·chargeback_initiated/won/lost) · **`occurred_at`**(순서 역전 판단) · **`currency`**·**`amount_minor`**·**`consumer_country`**(VAT·환불 추적) · `pg_provider`.
- **신규 `billing_locks`**(customer_id, cycle_date): Cron 진입 시 INSERT OR IGNORE 잠금.
- **신규 `notification_log`**: 고지·통지 발송 멱등.
- **consents**: `type` 세분화(tos·privacy·auto_renew·data_transfer_abroad·withdrawal_waiver·price_change) + ip·ua·ts.
- **Upstash 키**: `blacklist:{google_sub}`(취소·환불·분쟁 즉시 무효화, TTL=남은 exp), `checkout_lock:{google_sub}`(중복 결제 방지, EX 600).

### K2. 갱신 스케줄러·dunning·grace [BLOCKER]
- 토스 빌링은 dunning 자동 미제공 → 판매자 필수 구현. Cron = **매일 UTC 00:00 단일 트리거**(특정일 지정 시 말일 누락 회피). 핸들러: `WHERE DATE(expires_at) <= 오늘 AND status='active'` → 빌링 API(`idempotencyKey={sub}_{YYYY-MM}`) 청구. 1만명은 100명 배치(cursor를 KV 저장) 또는 QStash fan-out(자동 재시도 우위).
- **grace 3~7일**: 실패 시 `status='past_due'`·failed_count++. **에러코드 분기** — INVALID/EXPIRED_CARD = 재시도 무의미 → 즉시 grace + 카드 재등록 이메일. 일시 실패(잔액·한도)만 D+1/D+3/D+6 재시도. grace 만료 + 재시도 소진 시 tier='free' + 빌링키 폐기.
- **JWT-grace 정합**: 발급 시 `effective_tier = (tier='pro') OR (grace_until > now) ? 'pro' : 'free'`(grace 중 pro 유지).
- 말일: `expires_at = MIN(가입일 day, 해당월 말일)`.

### K3. 구독 상태 FSM + webhook 신뢰성 [BLOCKER]
- **FSM 명문화**: free → active → (past_due → grace) → canceling → expired, + suspended(분쟁)·refunded. tier와 subscription_status는 **단일 트랜잭션 동시 갱신**(불일치 방지).
- **순서 역전**: upsert에 occurred_at 비교(`WHERE expires_at IS NULL OR < occurred_at`). canceled가 paid보다 먼저 오면 pending_cancel 보관 후 규칙 적용.
- **최종 유실 복구**: 포트원 재시도 소진 대비 **reconciliation Cron(10분)** = 포트원 결제 단건조회 API로 미반영 결제 동기화. HMAC 실패 = 400(재시도 유도). 동일 event_id 재수신 시 UPDATE 조건 명세.
- **chargeback**: 수신 즉시 `suspended` + 운영 알림. 포트원 chargeback relay 여부 [확인필요], 미지원 시 PG 직결 이메일 알림.

### K4. 인증·세션 보강
- **JWT exp 5~15분 + silent refresh**(만료 2분 전 자동 재발급) — 취소·신규 반영 지연 최소화. **블랙리스트 필수화**(환불·분쟁 시), 키 = google_sub, jti 포함.
- **키 로테이션**: `kid` 클레임 + `JWT_SECRET_CURRENT`/`PREV` 이중 검증, 72h 후 prev 삭제(전원 무효화 회피).
- **getAuthToken**: 결제 연결은 **interactive:true**(계정 선택 강제 — Chrome Sync 기본계정 오선택 방지), 실패·기업계정 차단 시 재시도 UI + `launchWebAuthFlow` 대안 [확인필요]. **email 스코프** 추가. MV3 서비스워커 토큰 만료 시 재인증 흐름(무음 실패 → 갑작스런 free 표시 방지).

- google_sub 보관 근거·기간·탈퇴 시 D1 행 삭제 절차(개인정보보호법).

## L. 로그인·인증 재결정 (2026-06-29 — 브라우저 계정 의존성 조사)

> 4각도 웹조사(run wf_c6eb5d27-7de)로 `chrome.identity getAuthToken`의 브라우저 계정 의존성을 검증. **결론: getAuthToken 폐기.** 아래가 §A2·§3·§K4의 인증 부분을 **대체**한다.

### L1. getAuthToken 폐기 근거 (4각도 만장일치)
Chrome Sync 기본 계정 암묵 종속(구독 주체 오선택) · 다중계정 버그(crbug/1173583) · **기업 Workspace 관리자 정책 차단**(admin_policy_enforced 400) · MV3 서비스워커 토큰 캐시 소멸 후 **무음 실패**(pro gate 불안정) · **Chrome 전용**(Brave·Firefox 미지원) · **Google 계정 삭제 → orphan 구독**(결제 회수·복구 불가) · tokeninfo는 프로덕션 비권장(debug 전용). 유료 구독에 부적합. 출처: developer.chrome.com/docs/extensions/reference/api/identity · reco.ai(admin_policy_enforced) · Bugzilla #1305421/#1635344 · developers.google.com/identity(tokeninfo).

### L2. 결정 — 내부 계정 + 자체 JWT (브라우저·Google 비종속)
- **계정 식별자 = 내부 `user_id`(D1), 이메일로 소유**. Google sub 직접 PK 금지(계정 삭제 시 orphan). → 결제 바인딩·크로스 디바이스·재설치·기기 이전·orphan 모두 해소.
- **front-door 확정: Google SSO 전용(`launchWebAuthFlow`)** — 명시적 "Google로 로그인" 버튼. **브라우저 로그인 상태와 무관하게** Google OAuth 페이지를 띄운다(`getAuthToken`의 암묵 계정 사용과 정반대 → Chrome 종속·Sync 오선택·기업 차단·orphan 없음). Worker가 code 교환·client_secret 보호. **이메일은 Google OAuth(email 스코프)에서 자동 수집** → 결제·법정 고지(K6)·복구·크로스디바이스용(별도 매직링크 불요). **당장은 Google 외 로그인 미제공**(매직링크·타 소셜 보류) — 같은 launchWebAuthFlow 패턴으로 추후 확장. 계정 = 내부 `user_id`(google_sub 매핑, 소셜은 입구).
- 흐름: 로그인(매직링크 또는 Google 소셜) → Cloudflare Worker가 검증 + 내부 user_id 매핑(D1) → **자체 JWT(§A4) 발급** → chrome.storage.local 저장 → API는 JWT 서명만 검증. **getAuthToken 완전 제거.**
- 대안: **Clerk**(@clerk/chrome-extension 공식 MV3 SDK · service worker 세션 갱신 · 50K MAU 무료 · @hono/clerk-auth networkless 검증) = 매니지드로 가장 빠름, 단 벤더 종속. **Better Auth**(Cloudflare D1 네이티브 · 셀프호스트 · 벤더 0). 셀프(매직링크) ↔ Clerk = 1인 운영부담 vs 이식성 트레이드오프.

### L3. 스키마·설계 영향 (K1·§A2·§H 갱신)
- **D1 users**: PK = `user_id`(내부) · **`email` UNIQUE**(소유 식별자) · `google_sub` NULL 허용(소셜 연결 보조 컬럼) · 나머지 K1 동일. **google_sub 단독 PK 폐기.**
- **§A2 대체**: getAuthToken → 이메일 매직링크 + (보조) launchWebAuthFlow 소셜. **§K4의 getAuthToken interactive·Sync 오선택·기업 차단 관련 항목은 본 결정으로 해소.** JWT exp 단축·silent refresh·키 로테이션·블랙리스트(§K4)는 그대로 유효.
- **§H 결제 여정**: "계정 연결(Google OAuth)" → **"Google로 로그인"(launchWebAuthFlow, 명시 버튼)**. 재설치·기기 이전 복구 = **Google 재로그인**(같은 google_sub → 같은 내부 user_id).
- 이식성: 인증은 adapters에만(매직링크·JWT = Worker 자체 구현, 벤더 0). Clerk 채택 시에도 adapters 한정.

### L4. 확인·결정 필요
- **front-door 확정(사용자, 2026-06-29): Google SSO 전용(launchWebAuthFlow), 매직링크·타 소셜 미제공(향후 확장).** Clerk 등 매니지드는 보류.
- **크로스 브라우저**: launchWebAuthFlow Google 로그인은 **Chromium 계열(Chrome·Edge·Brave) 동작**(getAuthToken은 Edge·Brave 불가 → launchWebAuthFlow가 정답, Edge는 99+ 크래시 수정). **Firefox는 Google 스코프 버그(#1635344) + 스토어 별도(AMO)** → 추가 작업, 현재 타깃 외. 구현 시 Edge·Brave 실측 확인.
- **명시 SSO 버튼**: 브라우저 미로그인 사용자도 launchWebAuthFlow가 Google 로그인 페이지를 띄우므로 별도 처리 불요(이게 getAuthToken 대비 핵심 이점).
- 이메일 인프라(Resend vs Cloudflare Email Workers)는 K6 고지용으로 별도 필요(로그인용은 아님, 이메일은 Google에서 수집).
