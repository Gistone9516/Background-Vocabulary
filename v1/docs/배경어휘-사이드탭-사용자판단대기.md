### 1. detail 출처 영어화 [품질]

- 옵션: detail 검색 쿼리를 영어 용어/분야로 구성, 또는 recommend의 영어 RAG 캐시를 구조화 저장해 재사용. 어느 방향?

### 2. WORKER_BASE 실 도메인 [배포]
- 현재 `api.ts`의 WORKER_BASE = `sidetab-api.example.workers.dev`(placeholder). 빌드(프로덕션)는 이 주소로 fetch하므로 실제로 동작하려면 배포된 워커 URL 필요. DEV(vite dev)는 mock이라 무관.
- 로컬 검증 단계: `wrangler dev`(보통 localhost:8787) + WORKER_BASE를 로컬로 + manifest host_permissions에 localhost 추가가 필요.

### 3. Pretendard 폰트 self-host [MV3 CSP]
- theme.css가 panel.html처럼 CDN(jsdelivr)에서 Pretendard를 불러옴. MV3 CSP가 외부 CDN 폰트를 막을 수 있음 → 프로덕션은 폰트를 확장에 포함(self-host)해야 안전. 지금은 차단 시 시스템 폰트로 폴백(렌더 자체는 정상).

### 4. 진입 context_object(붙여넣은 문서) 입력 [UX]
- 현재 진입 컴포저는 입력 한 칸 + "조건 추가"(user_condition)뿐. "막힌 문서를 붙여넣기"용 별도 context_object 필드는 없음(원문 입력이 topic으로 감). 문서 붙여넣기 UX를 따로 둘지?

### 5. "AI로 더 정리" 무료 클릭 동작 [UX 확인]

## 보안·비용 감사 보류(Tier3 로드맵) — 2026-06-28 A-Z 전수 감사

> 6렌즈 적대 검증(보안·상태·에러·계약·비용·UX) 확정 33건 중, 명확한 기능 버그와 싼 비용 백스톱은 이번에 수정했다. 아래는 **진짜 인증·결제 인프라(Tier3)가 필요해 애드혹 패치 시 정상 사용까지 깨질 위험**이라 보류한 항목과, 한 건의 결정 사항이다. 코드 자체가 이미 "설치 UUID 바인딩 Tier3 전까지 구조적 잔존"으로 명시한 영역이다.

### B1. 익명 주간 미터링 우회 [Tier3]
- 현재: `x-user-id` 헤더를 생략하면 `metered=false`라 주간 카운터를 타지 않는다(`index.ts` classify). 익명·비브라우저 호출이 per-user 한도를 통째로 비껴간다.
- 임시 방어: 전역 일일 캡(이번 감사로 `/next`·`/preview`·`/relate`에도 확장)과 IP 레이트리밋.
- 옵션: ① 익명에 IP 기반 주간 카운터 ② 설치 UUID 서버 바인딩(Tier3, 결제와 함께). 권장 = ②.

### B2. x-tier 헤더 미검증(누구나 paid 선언) [Tier3]
- 현재: `readTier`가 클라가 보낸 `x-tier`를 그대로 믿는다. free 사용자가 `x-tier: paid`로 높은 토큰 상한과 무제한 detail을 받는다.
- 임시 방어: 전역 일일 캡 + IP 레이트리밋(비용 총량은 막힘, 티어 구분만 우회 가능).
- 옵션: 서버 결제 검증(Tier3). 그 전까지는 무료 모델 BM이라 손해가 토큰 단가 차이로 제한적.

### B3. detail 무료 횟수 서버 카운터 없음 [Tier3·모니터링]
- 현재: `detailLimitFree`(3회) 검사가 클라 메모리에만 있다. 새로고침이나 직접 호출로 무력화 가능(detail은 Tavily+LLM 이중 비용).
- 임시 방어: 클라 가드 + 전역 일일 캡 + IP 일 200회.
- 옵션: IP+day detail 전용 카운터. 비용 실측 후 필요하면 추가.

### B4. CORS origin=null 와일드카드 [배포 시 잠금]
- 현재: Origin 헤더 없는 요청(curl 등)에 `*` 반환. 비브라우저 호출이 무제한 접근.
- 권장: 프로덕션 배포 때 `ALLOWED_EXT_ORIGIN`(확장 ID) 설정 + 개발 외 no-Origin 거부. 배포 가이드에 반영.

### B5. 고위험 게이트의 client domain_risk 신뢰(매핑 미스 도메인) [저위험·모니터링]
- 현재: 정적 도메인맵에 안 잡히는("other") 도메인은 클라가 보낸 `domain_risk`를 그대로 라우팅에 쓴다. 의료·법률처럼 들리지만 키워드를 피한 입력에 `domain_risk: low`를 실으면 고위험 거부를 우회.
- 완화: 정적 키워드맵(의료·진단·medical·법률·legal 등)이 1차 방어이고 서버측이라 우회 불가. 우회의 결과도 "어휘 추천"이라 피해 상한이 낮음.
- 옵션: 미매핑 도메인 서버 재분류 또는 추가 키워드 스크린. 모니터링하다 필요 시.

### 결정(보류 아님) — 전역 캡 fail-open 유지
- 항목: `bumpGlobalCap`·`rateLimited`가 Upstash 오류 시 통과(fail-open)라, Upstash 장애 동안 캡이 무력.
- 결정: **fail-open 유지**(fail-closed 미적용). 이유 = fail-closed는 Upstash 일시 장애(네트워크 blip)에 전 사용자 서비스를 중단시킨다(가용성 우선 설계 의도). CF 워커는 isolate가 휘발이라 in-memory fallback 카운터도 사실상 무효. 주요 비용 구멍(미집계 LLM 엔드포인트)은 이번에 닫았으므로 실효 위험이 낮다. 재검토 = 배포 후 비용·장애 빈도 실측 시.
