# 배경노트(Vock note) v2 소프트웨어 요구사항 명세서(SRS)

| 항목 | 내용 |
|---|---|
| 문서 상태 | 초안 v0.2 (4렌즈 검증 반영, 레드펜 대기) |
| 작성일 | 2026-07-20 |
| 대상 시스템 | 배경노트 v2: 웹사이트 + 데스크톱 프로그램(Windows, Linux 호환) |
| 입력 | v1 승계원장 정본 4종(01 기획·02a 추천계약·07 보안부채·13 제품근거) + 사용자 확정 결정(2026-07-20) |
| 추적성 | 각 요구사항의 근거 표기는 요구사항 후보 ID(예: 01a-FR-12)이며, 후보는 승계원장 항목으로, 원장은 v1 원본으로 역추적된다 |

## 1. 개요

### 1.1 목적
이 문서는 배경노트 v2가 구현해야 할 기능·품질·제약을 구현 착수 전에 확정한다. 아키텍처와 기술 선정(HOW)은 후속 설계 문서(ADR)의 몫이며, 이 문서는 무엇을(WHAT) 만들 것인가만 다룬다.

### 1.2 제품 한 줄 정의
막연한 생각은 있지만 그것을 설명하는 마찰이 큰 사용자가, 설명 없이 클릭과 재인만으로 자기 상황에 맞는 배경지식 어휘(말그릇)를 얻게 하는 도구. [13-FR-01]

### 1.3 범위
- 포함: v1(크롬 확장)이 검증한 전 기능의 승계(사용자 확정), 데스크톱 고유 기능(9절, 제안 상태), free/pro 티어 게이팅 구조.
- 제외: 결제 수단·가격·PG 선정(별도 재설계 문서), 마케팅·운영 조직, v1 코드의 이식 방법.
- v1과 다른 점: 폼팩터가 크롬 사이드패널(320px)에서 웹사이트와 데스크톱 앱으로 바뀐다. v1의 UI 배치 규칙 중 좁은 패널 전제인 것은 요구사항이 아니라 재설계 대상으로 분류했다.

### 1.4 용어
| 용어 | 정의 |
|---|---|
| 좁히기(narrow) | 내용형 다지선다를 반복해 사용자의 의중을 특정하는 대화 단계 (아키네이터식) |
| job_type | 사용자의 작업 유형. 고정 열거값 9종(6.2절) |
| gap_type | 어휘 막힘 유형. 고정 열거값 a~e 5종(6.3절) |
| user_condition | 사용자가 입력한 개인 조건(업력·수치·상황 등) |
| context_object | 사용자가 붙여넣은 원문 문서·수치 맥락 |
| 담기(keep) | 어휘를 자산으로 보존하는 행위 |
| 프라이머(primer) | 담은 어휘와 상황을 메인 AI에게 전달하는 정리 텍스트 |
| RAG | 검색 결과를 근거로 주입하는 생성 방식. 전 티어 필수(5.2절) |
| 티어 | free / pro 두 등급. 서버가 판정하며 클라이언트 신고를 신뢰하지 않는다 |

## 2. 제품 개요

### 2.1 핵심 축 (변경 불가)
제품의 유일한 기획 축은 표현(articulation) 비용 면제다. 사용자는 어느 단계에서도 자기 상황을 길게 설명하도록 강요받지 않아야 하며, 모든 UI·기능·카피 판단의 정렬 기준은 "이 기능이 표현을 강요하는가"이다. [13-FR-01, 13-C-05]

과거 한때 기둥으로 잘못 기록된 "de-anchoring(넓힌 뒤 좁힌다)"류 논리는 v2 기획·요구사항에 반영을 금지한다. 정정된 실체(2026-07-20): 이는 사용자 인지 확장 이론이 아니라 이전 세대 AI의 입력 앵커링 편향을 우회하던 개인 프롬프팅 요령(넓은 주제를 먼저 물어 AI가 더 넓은 스코프를 재참조하게 함)이었고, 개인 요령의 기능화 금지(C-03)와 같은 뿌리다. 금지 이유는 아이디어의 오류가 아니라 제품 기획 목적이 아니기 때문이다. [13-C-03, 13-L-01, 사용자 정정 2026-07-20]

### 2.2 타깃
"그 분야를 모르는 사람"이 아니라 "막연한 생각은 있는데 설명하는 마찰·거부감이 큰 사람". 핵심층: 마감 있는 분야 진입자와 면접·이직 준비자, AI로 만드는 1인 빌더와 인디 메이커, 도메인 확장 직장인. 요구사항 우선순위는 이 타깃 정의를 벗어나지 않는다. [13-C-06]

### 2.3 플랫폼 (2026-07-20 사용자 확정)
- 웹사이트: 주 접점. v1 전 기능을 제공한다.
- 데스크톱 프로그램(Windows, Linux 호환): 웹과 같은 기능에 더해 데스크톱 고유 기능(4.9절)을 추가한다.
- v1의 크롬 확장 선택 근거(웹 결제 수수료 회피, 페이지 자동읽기)는 이 결정으로 무효화되었으므로 어떤 요구사항도 그 근거를 전제하지 않는다. [01c-CONST-09]

### 2.4 사용자 여정 요약
진입(상황 칩 클릭 또는 짧은 자유 입력) → 좁히기(내용형 다지선다 3~8턴) → 난이도 선택 → 어휘 리스트(우선순위 5~8개, 스트리밍) → 상세 열람·태깅·담기 → 정리(프라이머)·복사 → 세션과 어휘 자산으로 영속. [01b-FR-44]

## 3. 제약 (요구사항에 우선하는 상위 규범)

### 3.1 제품 정체성 제약
- C-01. 제품의 알맹이는 앞단 인터랙션 레이어이며, 뒷단 검색·사전 엔진이 아니다. 뒷단이 교체되어도 인터랙션 레이어 설계는 재사용 가능해야 한다. [01a-CONST-02, 13-C-04]
- C-02. 외부 검색(RAG)을 핵심 차별점 서사로 내세우지 않는다. 구현 디테일로만 취급한다. [13-C-01]
- C-03. 사용자 개인의 업무 요령(메인 AI에게 상위주제 배경을 먼저 정리시키는 방식)을 제품 기능으로 만들지 않는다. 주의: FR-604의 프라이머는 이 금지의 대상이 아니다. 프라이머는 사용자가 직접 획득한 어휘·상황을 맥락으로 넘기는 것이고, 금지 대상은 "메인 AI에게 배경 정리를 대신 시키는" 요령의 기능화다. [13-C-02]
- C-04. 제품 목표는 분야 마스터가 아니라 번역 준비다. 상황 맞춤 우선순위 어휘를 골라 쥐여주는 것이지 전체 나열이 아니다. [01a-CONST-03]
- C-05. 종착점은 사용자의 이해이지 복사용 프롬프트 생성이 아니다. 성공 지표를 paste율 등 대리지표로 대체하지 않는다. [01a-CONST-04, 01a-LESSON-02]
- C-06. 사용자는 항상 비전공자로 기본 가정하고, 배경이 확인될 때만 눈높이를 상향한다. [01a-CONST-01, 01c-CONST-02]

### 3.2 서비스 범위 제약
- C-07. 서비스 대상은 저위험 도메인(기술·창작·비즈니스 등)의 학습 맥락으로 한정한다. 의료·법률 등 개인의 진단·판단이 필요한 고위험 도메인과 캐주얼 일상 어휘는 지원하지 않는다. [01c-CONST-04]
- C-08. 계산·시뮬레이션·비교표 작성은 수행하지 않으며, 그 범위를 사용자에게 명시적으로 안내한다. [01a-CONST-11]
- C-09. 이미지 문서의 자동 OCR은 1차 범위에서 제외한다(데스크톱 폼팩터에서의 재검토는 열린 질문 9절). [01a-CONST-12]

### 3.3 언어·표현 제약
- C-10. 좁히기 선택지 레이블에 분야 전문어를 쓰지 않는다. 전문어는 내부 필드(domain_tags)에만 담는다. [01a-CONST-07]
- C-11. 실제로 구현하지 않은 정밀 방법론의 이름(EIG 등)을 사용자 대상으로 표방하지 않는다. [01a-CONST-10]
- C-12. RAG 검색 결과의 원문을 그대로 복사하지 않는다. 반드시 재서술하고 출처를 표기한다. 표기 형식과 발췌 한도는 법무 확인 후 확정한다(열린 질문). [01c-CONST-05, 01b-제약-11]

### 3.4 법규 제약 (대한민국 기준)
- C-13. AI 생성물임을 고지한다(AI 기본법, 2026-01-22 시행 생성물 라벨링 의무). [01c-CONST-06]
- C-14. 유료 서비스 운영 시 통신판매업 신고 의무를 이행한다. [01c-CONST-07]
- C-15. 개인정보처리방침을 게시한다. [01c-CONST-08]

### 3.5 설계 제약 (유지보수성, 사용자 전역 지시 2026-07-20 반영)
- C-16. 도메인 로직(core, shared에 해당하는 계층)은 특정 런타임·저장소·공급자 전역을 직접 참조하지 않는다. 외부 자원은 인터페이스(포트) 뒤로 추상화하고 주입한다. 이 경계를 깨는 설계 변경을 금지한다. [02a-CON-02, 02a-CON-03, 02a-CON-05, 02a-CON-06]
- C-17. 공용 계층은 웹표준 API만 사용한다(fetch, Web Streams, Web Crypto, AbortController 등). [02a-CON-04]
- C-18. LLM 호출은 벤더 중립 호환 레이어로 추상화해 모델 교체(fallback 포함) 비용을 최소화한다. [01b-NFR-08]
- C-19. API 키는 프론트엔드에 노출하지 않으며 서버 경유로만 LLM·검색을 호출한다. [01b-NFR-09]
- C-20. 데이터 스키마(Choice, Term 등)와 열거값(job_type, gap_type)은 단일 출처(SoT) 문서로 관리하며 구현자가 임의 변경할 수 없다. 요구사항은 선언에 그치지 않고 반드시 계약 문서(데이터 모델·API·화면 흐름)까지 관통시킨다. [01b-제약-01, 01b-제약-02, 01c-LESSON-01]
- C-21. 티어 판정 권위는 서버에만 있다. 클라이언트가 보낸 티어 신고 헤더를 신뢰하는 코드를 금지한다. [02a-CON-12, 07-NFR-06]
## 4. 기능 요구사항 (FR)

표기: FR-번호. 요구문 [근거 후보 ID] (필요 시 폼팩터 주석). "시스템은 ~해야 한다"는 필수, "~할 수 있다"는 선택 기능이다. v1 후반에 추가되어 정본 원장 밖에 있는 기능은 근거를 [v1 최종: 09·메모리]로 표기했고 레드펜에서 확정한다.

### 4.1 진입·입력 (FR-100대)
- FR-101. 시스템은 진입 화면에서 상황 칩(분야명이 아닌 일상어 상황 문장 레이블) 클릭을 주 경로로 제공해 분야와 job_type을 함께 포착해야 한다. [01a-FR-01, 13-FR-02] (칩 배치·수량은 넓은 화면 기준 재설계)
- FR-102. 진입 경로는 표현 0 상태로 시작해야 하며, 큰 자유서술 입력칸이 주 경로가 되어서는 안 된다. [13-FR-02]
- FR-103. 시스템은 짧은 자유 입력(하려는 말, 분야명, 붙여넣은 원문)에서 분야와 job_type을 추론하고, 붙여넣은 원문은 context_object로 보존해야 한다. [01a-FR-02]
- FR-104. context_object는 분야 추론 후 폐기하지 않고 좁히기 선택지 생성, 어휘 추천, 정리 생성까지 전 단계로 이월되어야 한다. [01a-FR-42, 01c-FR-04]
- FR-105. 시스템은 진입 단계에서 선택적 보조 조건 입력(관련 조건 한 줄)을 제공하고 user_condition으로 보존해 추천부터 반영해야 한다. [01a-FR-04]
- FR-106. 조건 의존도가 높다고 판정된 입력(condition_required=true)에서는 조건 입력 안내를 "선택"에서 "권장"으로 승격해야 한다. [01a-FR-05, 01a-FR-24]
- FR-107. 시스템은 입력의 조건 의존도 판정 결과를 condition_required(참/거짓)로 산출해야 하며, 판정은 고정 도메인 목록 방식이어서는 안 된다(목록 밖 도메인의 체계적 누락 방지. 판정 메커니즘 지정은 설계 몫, 고정 화이트리스트 금지는 설계 제약). [01a-FR-24, 01a-CONST-09]
- FR-112. user_condition을 두 채널(진입 조기입력, 정리 단계의 "내 상황 추가")에서 수집한 경우, 뒤 단계 최종값이 앞 단계 값을 덮어쓰고 앞 단계 값은 뒤 단계 입력란에 미리 채워(prefill) 제시되어야 한다. [01a-FR-25, 01b-FR-41]
- FR-108. 이미지 문서 입력에는 자동 OCR 대신 "핵심 수치·항목을 한 줄로 적어주세요" 수동 입력 유도를 제공해야 한다. [01a-FR-03, 01c-FR-17]
- FR-109. 시스템은 위험 분야 입력을 감지하면 이유를 설명하고 다른 맥락의 재시도를 유도하는 방식으로 부드럽게 거부해야 한다(확정 문구는 02a-FR-19의 정본 유지, 다국어 번역본 포함). [02a-FR-19]
- FR-110. 예시 칩 세트는 특정 주류 도메인에 편중되지 않고 다양한 상황을 포괄해야 한다. [01c-FR-08] (v1 최종: 4언어 각 약 150개 광도메인 [09·메모리])
- FR-111. 시스템은 텍스트 파일 첨부로 맥락을 입력하는 경로를 제공할 수 있다(v1 최종: pro 전용 텍스트 파일 첨부 [09·메모리], 티어 배치는 8절).

### 4.2 좁히기 (FR-200대)
- FR-201. 시스템은 내용형 3~4지선다(일상어 상황문 레이블)로 의중을 좁혀야 하며, 첫 분기는 작업 유형을 가르는 질문으로 고정해야 한다. [01a-FR-08]
- FR-202. 좁히기는 3턴 수렴을 목표로 하되 enough·confidence 신호로 조기 종료하고, 최소 3회 전에는 계속 진행하며 8회 도달 시 강제 종료해야 한다(엔벨로프 [3,8]). 질문 수를 채우기 위한 임의 연장은 금지한다. [01b-FR-27, 02a-FR-02]
- FR-203. 입력이 이미 구체적이면(비교 대상·수치·조항, 또는 어휘 직접 명시) 좁히기를 단축·생략하는 fast-track을 제공해야 한다. [01a-FR-11, 01c-FR-16]
- FR-204. 사용자는 "지금 충분해요" 조작으로 서버 판단과 무관하게 즉시 추천 단계로 넘어갈 수 있어야 한다(기획 초기의 "지금 정리 보기" 상시 버튼은 v1 최종에서 이 조작으로 흡수됨). [02a-FR-05, 01a-FR-09]
- FR-205. 복수 의중이 감지되면 다중 선택(최대 2개)을 허용하고 두 갈래를 함께 반영해야 한다. 단일 선택으로 강제 수렴시키지 않는다. [01a-FR-22, 01c-FR-09]
- FR-206. 되돌리기(1단계)를 지원하고, 한 번의 선택으로 후보 의중을 0으로 소거하지 않는 확률적 갱신을 사용해야 한다. [01a-FR-18] (v1 최종: 되돌리기 1회 사용 후 비활성 [09·메모리])
- FR-207. 선택·되돌리기 이력은 {label, action} 형태로 누적 기록되어 다음 질문 생성에 반영되어야 한다. [02a-FR-04, 01a-FR-41]
- FR-208. 신규 선택지의 domain_tags가 직전 2개와 절반 이상 겹치면 재생성(최대 2회)하고, 그래도 겹치면 다른 태그를 강제한 선택지로 대체해야 한다. [01a-FR-16]
- FR-215. 시스템은 반대 방향의 실패(연속 선택 중 도메인 태그가 맥락 없이 이탈하는 drift)도 감지하는 자동 가드를 두어 선택지 품질을 점검해야 한다. [01b-FR-50]
- FR-209. 같은 분야의 첫 1~2개 분기 선택지는 캐시해 모든 사용자에게 동일하게 제공해야 한다. [01a-FR-17]
- FR-210. 시스템은 "선택지가 어려워요" 조작을 제공하고, 선택 시 이후 턴의 선택지 난이도를 낮춰야 하며 이 조작은 턴을 차감하지 않아야 한다. [v1 최종: 09·메모리]
- FR-211. 선택 칩과 직접 입력을 함께 쓸 수 있는 경우 두 입력의 합산이 보장되어야 한다. [v1 최종: 09·메모리]
- FR-212. 좁히기 진행 표시는 목표 턴 기준의 연속형이어야 한다(턴이 연장되어도 점이 무한히 늘어나지 않는 형태). [01b-FR-28]
- FR-213. LLM 응답 대기 중에는 이전 화면 내용을 유지한 채 조작만 비활성화하고, 2초 초과 시 진행 중 안내를 인라인으로 표시해야 한다(레이아웃 점프 금지). [01a-FR-44]
- FR-214. 비전공·비전문 직군이 감지되면 선택지 언어를 그 역할의 시선에 맞춰 조정해야 한다(합격 판정은 NFR-203 수동 게이트 루브릭). [01b-FR-25]
- FR-216. 사용자 배경 수준 추론이 모호할 때만, 좁히기 중 1회에 한해 배경을 묻는 능동 질문으로 보강해야 한다(기본은 입력에서의 수동 추론, 비전공자 기본값). [01a-FR-23]

### 4.3 난이도 선택 (FR-250대)
- FR-251. 좁히기 종료 직전 난이도(기초/중급/심화)를 고르는 단계를 제공해야 하며, 좁혀진 주제의 깊이별 대표 어휘 예시(어휘+한 줄)를 보여줘야 한다. [v1 최종: 09·메모리]
- FR-252. 난이도 예시는 세션과 답변 조합으로 캐시해 재진입 시 재생성하지 않아야 한다. [v1 최종: 09·메모리]

### 4.4 어휘 추천 리스트 (FR-300대)
- FR-301. 시스템은 사용자 상황 최우선 추천 어휘 5~8개를 우선순위 순으로 제공해야 하며, 각 어휘에 종류 배지(kind)·근거(why)·그룹 칩을 포함해야 한다. why는 사용자의 목표·조건과 직결되어야 한다. [01a-FR-27, 01a-FR-46, 01c-FR-12]
- FR-302. 추천은 term 단위 스트리밍으로 도착 즉시 표시되어야 하고, 신규 항목은 시각적으로 구분되어야 한다. [02a-FR-06, 02a-FR-08]
- FR-303. 추천 어휘는 사용자가 입력에 직접 쓴 용어(앵커)를 제외하고, 앵커 주변의 실용·방법론 어휘를 골라야 한다. 각 정의는 짧고 쉬운 한 문장이어야 한다. [01b-FR-31, 01c-FR-21]
- FR-304. "어휘 더 보기"는 이미 노출됐거나 "알아"로 태깅된 어휘를 exclude로 전달해 중복 없이 다음 우선순위 묶음을 추가 생성해야 한다. [01a-FR-31, 01b-FR-18, 02a-FR-07]
- FR-305. 어휘 누적에는 티어별 상한(maxTotal)이 있어야 하며, 상한 도달 시 추가 생성을 차단하고 free 티어에는 업그레이드 유도를 표시해야 한다. [v1 최종: 09·메모리, 02a-FR-18 유형]
- FR-306. 그룹(상위 클러스터) 단위 보기 전환과 그룹 지정 추가 생성을 제공해야 한다(그룹별 생성 개수는 티어 차등). [01a-FR-30, 02a-FR-09, v1 최종: 09·메모리]
- FR-307. 조건 재탐색: 사용자가 조건을 덧붙여 재탐색하면 이를 새 답변 턴으로 처리해 좁히기를 이어가고, 생성 결과는 기존 리스트에 append해야 한다(초기화 금지). [v1 최종: 09·메모리] (v1은 pro 전용, 티어 배치는 8절)
- FR-308. 추천 단계에서 서버는 LLM 판정 domain_risk로 고위험 여부를 재차단해야 한다(입력 필터와 별개의 2차 게이트). [02a-FR-11]
- FR-309. 고비용 도메인(법령·세무 등 경계 사례)의 어휘 카드에는 "메인 AI나 전문가로 최종 확인" 경고를 표시해야 한다. [01b-FR-49]
- FR-310. 진행 중인 추천 스트림은 취소 가능해야 하며 취소 신호가 LLM 계층까지 전파되어야 한다. [02a-FR-12]
- FR-311. 리스트에는 범위 안내(계산·시뮬레이션은 메인 AI로)를 명시해야 한다. [01a-FR-34]

### 4.5 어휘 상세 (FR-400대)
- FR-401. 어휘 상세는 클릭 시점에 온디맨드로 불러와야 하며(전체 미리 생성 금지), 펼친 순간부터 스트리밍으로 채우고 한 번 연 내용은 캐시해야 한다. [01a-FR-47, 01b-FR-12]
- FR-402. 상세는 개념(what)·내 맥락(whymine)·활용(how)의 3단 구조와 선택적 보조 메모, 함께 볼 관련 어휘, 참고 출처로 구성되어야 한다. [02a-FR-13, 01a-FR-29]
- FR-403. 상세의 출처는 사이트명·제목·URL 1~3개를 표기하고, 출처가 없으면 "확인된 출처 없음, 모델 일반지식 기반"으로 명시해 신뢰도가 낮음을 알려야 한다. 출처 귀속이 애매한 경우 출처 배열을 비워 두어야 하며 거짓 귀속을 표시해서는 안 된다. [01a-FR-52, 02a-FR-14, 02a-CON-10]
- FR-404. 관련 어휘 클릭 시 해당 어휘의 즉석 상세 카드를 리스트에 삽입해야 한다(무반응 금지). [v1 최종: 09·메모리, 01b-FR-13의 점프를 v1이 삽입형으로 구체화]
- FR-405. gap_type과 job_type 조건에 따라 상세·카드의 조건부 필드(direction, use_example, context_note, relates_to, order)가 활성화되어야 한다(활성 규칙은 6.3절). [01a-FR-48~51, 01b-FR-02~08]

### 4.6 태깅·지식 상태 (FR-500대)
- FR-501. 각 어휘에 알아/몰라/적용모름 태깅을 지원해야 하며, 태깅 전 상태는 미확인으로 별도 취급해야 한다(미확인은 서버로 전송하지 않음). [01a-FR-32, 02a-FR-10, 02a-FR-20]
- FR-502. 지식 상태(알아/몰라/적용모름/미확인)는 세션 간 영속되어야 하고, 알아 한 어휘는 추천에서 제외되며 몰라 한 어휘는 복습 우선으로 노출되어야 한다. [01a-FR-40, 01a-FR-43, 02a-FR-28] (저장 방식은 v2 재설계: 계정 연동 서버 저장 또는 로컬, 열린 질문 9절)

### 4.7 담기·정리·내보내기 (FR-600대)
- FR-601. 사용자는 어휘를 담기(보존)할 수 있어야 하며, 담은 어휘는 어휘 자산으로 영속되어야 한다. [13-FR-03]
- FR-602. 기본 정리는 클라이언트에서 즉시, 추가 비용 없이 생성되어야 한다(태깅 결과 기반 템플릿). [02a-FR-16]
- FR-603. "AI로 더 정리"는 어휘·태그·조건을 반영한 개인화 정리를 생성해야 한다(v1은 pro 전용, 티어 배치는 8절). [02a-FR-17]
- FR-604. 프라이머(정리 텍스트)는 고정 템플릿("나는 [분야]에서 [task_intent]를 하려 한다...")으로 생성하되 값 없는 슬롯은 생략하고, 복수 job_type이면 두 작업을 함께 서술해야 한다. 프라이머 문구는 메인 AI에게 어휘를 다시 설명하라고 지시하는 문장을 포함해서는 안 된다(어휘는 이미 아는 맥락으로 전제). [01b-FR-32, 01b-FR-33, v1 최종 교정: 09·메모리]
- FR-605. 정리 결과의 복사와 공유(URL 인코딩 링크)를 제공해야 한다. 메인 AI로의 자동 주입은 범위 밖이다(deep link 도입 여부·방식은 Q-05로 이관, 이 FR의 인수 범위는 복사·공유만). [01a-FR-37, 01b-FR-45]

### 4.8 세션·프로젝트·영속 (FR-700대)
- FR-701. 좁히기 진행 상태는 매 턴 저장되어 크래시·이탈 후에도 이어서 진행할 수 있어야 한다. 세션별 턴 예산은 세션에 귀속된다. [v1 최종: 09·메모리]
- FR-702. 생성된 어휘 리스트는 담기 0개여도 세션에 보존되어, 이전 탐색 카드에서 클릭 시 전체 리스트를 복원할 수 있어야 한다. [v1 최종: 09·메모리]
- FR-703. 세션 목록은 검색(주제·영역), 고정(핀), 날짜 버킷, 삭제 실행취소를 지원해야 한다. [v1 최종: 09·메모리]
- FR-704. 프로젝트(폴더)로 세션을 묶을 수 있어야 하며, 프로젝트 진입 시 그 프로젝트 스코프의 홈(스코프홈)을 제공해야 한다. 세션 선택 UI는 화면 전체를 덮지 않는 플로팅 드로어 형태를 유지한다(사용자 명시 기획. 관찰 가능 기준 = 화면 전체를 덮지 않음, 최대 크기 정량 기준은 설계 단계 확정. 넓은 화면 재검토는 레드펜). [v1 최종: 09·메모리, 09-42]
- FR-705. 같은 프로젝트에 충분한 담은 어휘가 쌓이면, 좁히기 종료 직전 기존 자산과의 연결을 묻는 재인 질문(연결 턴)을 조건부로 제공해야 한다. 관련 없음 탈출과 실패 시 우아한 생략을 보장한다. [v1 최종: 09·메모리]
- FR-706. 프로젝트의 담은 어휘는 신규 추천에서 중복 제외(dedup)되어야 한다. [v1 최종: 09·메모리]

### 4.9 데스크톱 고유 기능 (FR-900대) — 전건 [제안], 레드펜에서 확정
- FR-901(제안). 데스크톱 앱은 로컬 파일(문서)을 드래그·선택으로 첨부해 context_object로 쓸 수 있어야 한다. 웹보다 넓은 파일 형식을 지원한다.
- FR-902(제안). 데스크톱 앱은 저장된 세션·어휘 자산을 오프라인에서 열람할 수 있어야 한다(생성 기능은 온라인 전용). ★Q-07(저장 위치) 결정에 조건부: 서버 저장 전용으로 결론 나면 이 FR은 로컬 캐시 요구로 재정의 필요.
- FR-903(제안). 데스크톱 앱은 전역 단축키로 빠른 입력창을 호출하는 퀵 캡처를 제공할 수 있다. [가정: Linux 일부 환경(Wayland 계열)은 전역 단축키 제약이 있을 수 있음 — 설계 단계에서 웹 근거로 확인]
- FR-904(제안). 데스크톱 앱은 OS 알림으로 복습 알림을 제공할 수 있다(설정에서 끄기 가능). [v1 흔적: 복습 알림 설정 존재]
- FR-905(제안). 데스크톱 앱은 자동 업데이트를 지원해야 한다(NFR-604의 서명·무결성 검증과 쌍으로만 유효 — 검증 없는 자동 업데이트 금지).

### 4.10 온보딩·다국어 (FR-950대)
- FR-951. 첫 사용자에게 4스텝 이내의 튜토리얼을 제공해야 한다(1회 노출, 건너뛰기 가능). [v1 최종: 09·메모리]
- FR-952. UI와 생성 출력은 한국어·영어·일본어·중국어를 지원해야 하며, 언어 설정은 영속되어야 한다. 위험 분야 거부 문구 등 정책 문구도 전 언어 정본을 갖는다. [v1 최종: 09·메모리, 01c-NFR-09]
- FR-953. 시스템은 한국어 텍스트 생성 시 받침에 따른 조사(을/를 등)를 정확히 선택해 생성해야 한다. [07-NFR-03]
- FR-954. 랜딩·온보딩은 "AI 시대에 배경지식을 알아야만 하는 이유"를 주장과 근거로 사용자를 설득하는 콘텐츠를 포함해야 한다. 근거는 웹 조사로 확보한 출처 기반이어야 하며(추정·과대주장 금지, 13-L-03의 근거강도 규율 적용), 정본은 별도 설득 논거 문서로 관리한다. [사용자 지시 2026-07-20, 조사 진행]
## 5. 비기능 요구사항 (NFR)

### 5.1 성능
- NFR-101. 좁히기 선택지 생성의 TTFT 상한을 정량 게이트로 두어야 한다. 확정 수치는 모델 선정 후 상세설계에서 정하되, 확정 전 잠정 게이트는 한국 기준 1.5초로 한다(v1 근거 승계). 게이트 초과가 반복 관측되면 대체 모델 전환(fallback) 경로가 있어야 한다(관측 기간·빈도 기준은 상세설계 확정). [01b-NFR-05]
- NFR-102. 어휘 추천은 term 단위 스트리밍으로 제공해야 하며, 첫 term 표시까지의 시간에 정량 게이트를 둔다(확정 수치는 상세설계, v1 실측 참고: TTFT 약 0.8초). [01c-NFR-02, 01c-NFR-03]
- NFR-103. 2초 초과 대기에는 반드시 진행 중 피드백을 제공한다(FR-213과 연동). [01a-NFR-04]

### 5.2 품질(정확도)
- NFR-201. RAG는 전 티어 필수다(v1 실측: 무RAG 정확도 68~75로 부족, RAG 적용 84.6~86.2). [01a-NFR-01, 01c-NFR-01]
- NFR-202. 채택 모델·설정 조합은 대표 시나리오 out-of-sample 평가를 정량 게이트로 통과해야 한다. 게이트 수치·채점 척도·평가셋 크기는 모델 선정 후 상세설계에서 확정하며, 확정 전 잠정 기준은 v1 방식(100점 척도 85점 또는 상위 모델 대비 15% 이내)을 따른다. v2가 모델을 재선정하면 수치·레시피는 재실측한다(v1 수치 승계 금지). [01a-NFR-02, 교훈 B-42]
- NFR-203. 출시 전 대표 고난도 분야 5~6개로 선택지·추천·조건부 필드의 수동 품질 평가 게이트를 거쳐야 하며, 평가 분야는 실제 서비스 대상 시나리오와 일치해야 한다. 채점 루브릭(합격선 포함)은 상세설계에서 별첨으로 확정하고, 형용 요구(FR-214, FR-301, FR-303, FR-110 등)의 판정도 이 루브릭에 위임한다. [01a-NFR-06, 01c-LESSON-03]
- NFR-204. 도메인 위험도에 따라 처리 경로를 라우팅한다(저위험은 경량 모델, 고난도는 상위 모델. 원리만 승계, 모델명은 설계 단계). [01b-NFR-07, 02a-NFR-06]
- NFR-205. 출시 전 exclude 기반 페이지네이션의 중복·고갈과 enough 신호의 조기·지연 종료에 대해 실측 캘리브레이션을 수행하고 임계값을 확정해야 한다(수행 여부가 게이트). [02a-NFR-03, 02a-NFR-04]

### 5.3 보안 (v1 보류 부채의 전면 해소, 처음부터 설계에 반영)
- NFR-301. 사용량 한도는 익명·비인증 요청에도 서버 측에서 강제한다(클라이언트 식별자 존재 여부에 의존 금지). [07-NFR-04]
- NFR-302. 전역 일일 캡과 IP 레이트리밋을 모든 비용 발생 엔드포인트에 적용한다. [07-NFR-05]
- NFR-303. 티어 판정은 서버 저장 구독 상태로만 하며(자체 발급 토큰의 클레임 포함), 클라이언트 자기신고를 신뢰하지 않는다. [07-NFR-06, 02a-FR-23]
- NFR-304. 무료 티어의 고비용 기능 사용 횟수는 서버 측 카운터로 강제한다. [07-NFR-07]
- NFR-305. 프로덕션은 허용된 클라이언트만 접근 가능해야 한다. 웹은 Origin 화이트리스트, 데스크톱 앱은 별도 클라이언트 인증 방식을 설계 단계에서 확정한다. [07-NFR-08]
- NFR-306. 고위험 분야 판정은 클라이언트 신호에만 의존하지 않고 서버 측 정적 키워드 매핑을 1차 방어로 유지한다. [07-NFR-09]
- NFR-307. 인증은 내부 계정(user_id 기본키, email 소유 판별) + 표준 OAuth 기반 Google SSO + 자체 JWT 원리를 승계하되, 구체 플로우는 웹·데스크톱 폼팩터로 재설계한다. [02a-FR-23, 02a-CON-07] (v1의 launchWebAuthFlow는 확장 전용이라 승계 금지)
- NFR-308. ★크랙 방지 원칙(사용자 지시 2026-07-20): 유료 기능의 실체(LLM 파이프라인, 티어 판정, 한도 카운터)는 전부 서버에서 실행·검증되어야 하며, 클라이언트(웹·데스크톱 바이너리)를 변조·크랙해도 유료 기능이 열리지 않는 구조를 유지해야 한다. 클라이언트에 유료 게이팅의 단독 권위를 두는 설계를 금지한다(NFR-303의 강화). 인수 기준: 클라이언트 코드만 수정해 pro 응답을 얻는 경로가 존재하지 않아야 한다.
- NFR-309. 데스크톱 바이너리의 변조 저항(무결성 검사, 토큰·시크릿의 안전한 로컬 보관)은 보조 수단으로 적용하되, 신뢰 경계는 항상 서버다. 구체 기법과 수준은 설계 단계에서 웹 근거로 확정한다. [사용자 지시 2026-07-20]
- NFR-310. 웹 클라이언트는 표준 보안 하드닝(CSP, XSS·CSRF 방어, 의존성 취약점 점검)을 갖춰야 하며, 구체 항목은 설계 단계 체크리스트로 확정한다. [사용자 지시 2026-07-20]

### 5.4 비용·가용성
- NFR-401. 같은 분야의 공통 고정 구간(첫 분기, 분야 지도, 도메인 근거 캐시)은 캐시해 반복 호출을 방지한다. [01b-NFR-02~04, 01c-NFR-04]
- NFR-402. 온디맨드 추가 생성(더 보기, 그룹 생성, 상세)은 무료 티어에 호출 한도를 둔다(구체 수치는 운영 설정으로 외부화). [01a-NFR-05, 01b-NFR-11]
- NFR-403. 운영비는 사용자 규모 대비 목표 수준 내로 관리한다(v1 참고치: 1만 사용자 월 $85~$204, v2 재산정 필요). [01c-NFR-05, 01c-NFR-06]
- NFR-404. 사용량 제한 저장소 장애 시 fail-open(가용성 우선)을 유지한다. 재검토는 배포 후 실측 시로 한정한다. [07-NFR-10, 07-CONSTRAINT-02]
- NFR-405. 모든 LLM 응답은 구조화 출력(JSON)으로 강제하고, 실패 대비 방어적 파서를 둔다. [01b-NFR-10]

### 5.5 운영·계측
- NFR-501. 환경(local/dev/prod)별 API 주소·설정은 분리 구성한다. [07-NFR-01, 07-CONTRACT-04]
- NFR-502. 웹은 외부 폰트 로드 실패 시 시스템 폰트로 폴백해 렌더링이 깨지지 않아야 하고, 데스크톱 앱은 폰트를 번들에 내장해 오프라인에서도 동일하게 렌더링되어야 한다. [07-NFR-02] (v1의 self-host 사유는 확장 CSP 전제라 재해석)
- NFR-503. 시스템은 선택·태깅 클릭의 타임스탬프(렌더링부터 클릭까지 경과)를 계측·저장할 수 있어야 한다(망설임 지표). [01b-NFR-01]
- NFR-504. 운영 한도값(주간 한도, 누적 캡, 그룹 생성 수 등)은 코드 하드코딩이 아니라 서버 설정으로 외부화한다. [v1 최종: 09·메모리]
- NFR-505. 빌드 프로세스는 도메인 계층(core, shared)이 런타임 전용 모듈을 import하거나 런타임 전역을 참조하면 빌드를 실패시키는 경계 테스트를 포함해야 한다(C-16의 기계 검증). [02a-NFR-01]

### 5.6 플랫폼 적합성 (v2 신설 — 폼팩터 전환으로 처음 필요해진 요구)
- NFR-601. UI는 뷰포트·창 크기 변화에 반응형으로 동작해야 하며, 최소 지원 창 크기와 브레이크포인트별 레이아웃 규칙(정량)은 상세설계에서 확정한다. 최소 지원 크기에서 기능 손실이 없어야 한다.
- NFR-602. 웹의 지원 브라우저·버전 매트릭스를 상세설계에서 확정하고, 매트릭스 내 전 조합에서 핵심 여정(2.4절)이 동작해야 한다(모바일 웹 대응 여부는 Q-13).
- NFR-603. 키보드만으로 핵심 여정(2.4절)을 완주할 수 있어야 하며, 명도 대비와 스크린리더 대응 수준(목표 기준)은 상세설계에서 확정한다.
- NFR-604. 데스크톱 앱의 설치 파일과 자동 업데이트 채널은 코드 서명과 업데이트 무결성 검증을 갖춰야 한다(FR-905의 전제 조건).

## 6. 데이터 요구사항 (요구 수준의 고정 어휘)

상세 스키마는 설계 문서로 인계하되(7절), 아래 열거값과 구조 원칙은 요구사항으로 고정한다.

### 6.1 핵심 데이터 단위
- Choice(선택지): 일상어 레이블 + 내부 전문어 태그(domain_tags). [01a-CONTRACT-01의 deeper 필드는 제외 — "더 깊이" 기능이 v1 최종에서 전면 제거됐으므로(결정 최신화 색인) 필드도 승계하지 않는다. 재도입하려면 제거 사유부터 확인]
- Term(어휘): 필수 = term, kind, group, priority, why, one_line, tag. 조건부 = direction, use_example, context_note, relates_to, order, gap_type. [01b-FR-40]
- 관계 개념 3종은 구분 유지: group(상위 클러스터), relates_to(함께 볼 동렬 어휘), related(상세 열람용 일반 관련어). [01b-FR-43, 01b-계약-10]

### 6.2 job_type 열거 (9종, 고정)
보고서작성, 의사결정, 서류제출, 이해학습, 진단판단, 문제해결, 협상설득준비, 전문가면담준비, 기획전략. 한 입력은 최대 2개. UI 레이블과의 매핑표는 설계 문서에 명시한다(임의 해석 금지). [01b-FR-22, 01a-FR-20, 01a-FR-21]

### 6.3 gap_type 열거 (a~e 5종, 고정) 와 필드 활성 규칙
a(뜻 모름)→one_line / b(관계·순서)→relates_to, order / c(재번역)→use_example, direction / d(수치·해석)→direction, context_note / e(조항 상호작용)→context_note, direction. 복합은 배열. Term별 지정 시 세션 기본값을 대체(합집합 아님). direction은 gap_type c·d·e 또는 job_type 의사결정·진단판단·협상설득준비의 합집합 조건. use_example 트리거 중복 시 재번역 예시 우선. [01a-FR-48~51, 01b-FR-02~08]

### 6.4 태그 상태
알아/몰라/적용모름 + 프론트 전용 미확인. 서버 전송은 3종만. [02a-FR-20]

## 7. 설계 단계 인계 목록 (SRS 범위 밖, 누락 방지용 색인)

다음은 HOW에 속해 SRS에서 확정하지 않으나, 설계(아키텍처·상세설계) 단계가 반드시 다뤄야 할 항목이다. 상세 후보는 srs-candidates의 "계약 후보" 절과 승계원장 02a에 있다.
1. 프롬프트 파이프라인 5종의 입출력 계약(v1 계약을 출발점으로 재설계). [01b-계약-03~07, 02a-CTR-07]
2. 엔드포인트 구성과 스트리밍 프로토콜, 취소 전파. [02a-CTR-06~07]
3. 저장소 전략: 어휘 자산·세션의 로컬/서버 배치(웹·데스크톱 공통 동기화 여부 포함). 미해결(9절).
4. 인증·엔타이틀먼트 데이터 모델(UserRecord, Entitlement, 토큰 정책)의 v2 재설계. [02a-CTR-11~18은 원리 참고]
5. 티어·모델 라우팅 매핑표. [02a-CTR-09]
6. 검색(RAG) 파이프라인과 도메인 근거 캐시, detail 출처 검색 전략(택1 미확정). [07-CONTRACT-06, 01c-CONTRACT-15]
7. 운영 설정 외부화 목록(한도·캡·플래그). [NFR-504]
8. 데스크톱 패키징·배포 형식(Windows 설치 파일, Linux 배포 형식 — 자동 업데이트·전역 단축키·샌드박스 권한을 좌우하는 선결 변수, FR-905·NFR-604 이전에 확정). [폼팩터 렌즈]
9. 딥링크 실현 수단(웹 프로토콜 등록, OS별 커스텀 프로토콜 핸들러 — Q-05 채택 시). [폼팩터 렌즈]
10. 접근성 구현 전략과 목표 기준(NFR-603 구체화). [폼팩터 렌즈]
11. 엔타이틀먼트 적용 연산의 상세 계약(멱등 upsert, occurred_at 순서 역전 방지 — TR-07의 구현). [02a-FR-25, 02a-FR-26]

## 8. 티어·게이팅 요구 (결제 재설계와 독립인 구조 요구만)

- TR-01. free/pro 2티어를 구분하고, 유료 전용 기능과 한도는 서버 검증으로 게이팅한다. [07-CONSTRAINT-01, 01c-FR-22]
- TR-02. free 티어에는 주간 탐색 한도를 두고, 차감 시점은 탐색 시작(실 LLM 호출 시점)으로 한다. 0답 이탈도 차감되며, 이어서 진행은 재차감하지 않는다. [v1 최종: 09·메모리]
- TR-03. 한도 소진·유료 기능 시도 시 업그레이드 유도 UI를 제공한다(다크패턴 금지). [01c-FR-23, 07-FR-03]
- TR-04. 결제 비활성 상태에서도 전원 pro로 취급하지 않고 평소 게이팅을 유지하며, 결제 진입점만 준비 중으로 표시한다. [02a-FR-22]
- TR-05. 구독 유예(grace) 중 사용자는 유효 권한상 pro로 취급한다. [02a-FR-24]
- TR-06. v1의 티어별 기능 배치(파일 첨부=pro, 조건 재탐색=pro, AI 정리=pro, 어휘 캡 8/32, 그룹 생성 2/4, 주 7회)는 참고 기본값이며, v2 결제 재설계 문서에서 최종 확정한다. [v1 최종: 09·메모리]
- TR-07. 엔타이틀먼트(구독 상태) 변경 적용은 멱등이어야 하며, 변경 발생 시각 기준으로 순서 역전(늦게 도착한 과거 이벤트가 최신 상태를 덮어쓰는 것)을 방지해야 한다. 결제 수단이 무엇이든 유효한 폼팩터 무관 요구다. [02a-FR-25, 02a-FR-26]
- TR-09. ★free 티어에는 광고를 표시한다(사용자 확정 2026-07-20). pro는 광고 없음(광고 제거가 pro 가치의 일부). 광고 네트워크·배치·정책은 결제·수익 재설계 문서에서 확정하되, v1의 "덜어내고 차분하게" 디자인 DNA와의 충돌을 설계 시 관리한다. 경위: v1의 광고 보류 결정 근거 중 "확장 게재 금지(AdSense)"는 웹사이트 전환으로 소멸했고, 사용자가 광고 BM을 확정으로 뒤집음. [v1 보류 결정: _참고/09, 재검토 트리거 충족]
- TR-08. 업그레이드 유도·구독 관련 UI에는 플랜 비교, 현재 사용량 잔여, 환불·약관 안내 링크가 포함되어야 한다(문구·가격·구성은 결제 재설계 문서 확정). [01c-FR-24]

## 10. 인수 기준 원칙

- 각 FR은 인수 시나리오를 작성할 수 있는 형태를 지향하되, 조작적 정의가 필요한 형용 요구(짧고 쉬운 한 문장, 다양성, 직결 등)는 상세설계에서 NFR-203 루브릭으로 판정 기준을 확정한다. 미확정 사안은 FR이 아니라 9절 열린 질문에만 둔다.
- v1 교훈에 따라(35시나리오 초기 통과 4/35) 시나리오 기반 반복 검증을 일정에 반영하고, 설계 정합 통과를 출력 정확도 검증으로 오인하지 않는다. [01c-LESSON-02, 01c-LESSON-04]
- MVP 검증 순서는 메타인지 → 의중 좁힘 → 전달 효용 → 존재이유 순을 유지하며, "메인 AI에 그냥 물어보면 되는데 왜 이걸 먼저 여는가"의 방어 명제는 여전히 미검증 가정으로 두고 v2에서 직접 검증한다. [01a-CONST-05, 교훈 B-32]
