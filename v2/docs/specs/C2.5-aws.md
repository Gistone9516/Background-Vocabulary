# C2.5 스펙 — aws 부트·배포 코드 (buildflow ②, SoT §0-2·§5·§6에서 파생)

> **배포 게이트 코드**: 로컬 스모크 불가(실 Aurora/Secrets 필요). 타입체크(@aws-sdk 실 타입)로 API-정확성만 확인. 실배포·스모크 = 핸즈온(`../../app/DEPLOY.md`).

## 0. 범위 / 비범위
- **범위(코드)**: `@vock/aws` — DataApiSqlRunner(SqlRunner)·Secrets 로더·buildAwsDeps·Lambda streamHandle 핸들러. migrate 문장 분리(Data API 다중문장 불가 대응, pg 공용).
- **비범위(핸즈온)**: 실 AWS 프로비저닝·배포·스모크(실 LLM 1회 = SoT §9 C2 완료 기준의 나머지 절반)·**IaC 도구 선택**(CDK/SAM/Terraform — 사용자 Cloud-OP 포트폴리오 결정). IaC 템플릿은 도구 확정 후 작성.

## 1. 핵심 설계 결정
### 1-1. DataApiSqlRunner로 리포 재사용(드라이버 교체)
- 리포는 SqlRunner 포트에만 의존하므로 aws는 DataApiSqlRunner만 주입하면 그대로 동작. `$n`→`:pn` 변환·Field 응답→row 매핑·트랜잭션(Begin/Commit/Rollback)·3분 상한 준수(중첩 트랜잭션 없음).
### 1-2. 콜드 스타트 1회 초기화
- handler.ts top-level await로 Secrets 로드 + 앱 조립 1회, Lambda 컨테이너 재사용. 핸들러=`dist/handler.handler`. 배럴은 handler 미노출(import 시 init 실행 방지).
### 1-3. streamHandle(RESPONSE_STREAM)
- `hono/aws-lambda`(별도 패키지 아님, hono 내장)의 streamHandle로 /recommend SSE 직렬화. §5: Lambda는 클라 끊김 미통지 → 서버측 취소 보완은 스트림당 상한(추후 실측 조정).
### 1-4. 서버리스 캐시·카운터 = Upstash 필수
- Lambda는 인메모리 비영속 → buildAwsDeps는 Upstash 필수(폴백 없음, deps에서 강제).
### 1-5. migrate 문장 분리(공용화)
- Data API는 다중문장 불가 → migrate가 `;` 단위 분리 후 개별 실행. pg도 동일 경로(단일문장 단순 프로토콜). 로컬 pg e2e 무회귀 확인.

## 2. 검증 상태
- **타입체크 통과**: DataApiSqlRunner·secrets·handler가 실 @aws-sdk/client-{rds-data,secrets-manager}·hono/aws-lambda 타입에 대해 컴파일(composite declaration emit = 전 타입 해소 강제). SDK 사용이 API-정확.
- 실호출 스모크는 핸즈온(DEPLOY.md 4~5절). 예상 보정 지점(jsonb typeHint·streamHandle 취소)은 DEPLOY.md에 명시.

## 3. 열린 항목 → 핸즈온
