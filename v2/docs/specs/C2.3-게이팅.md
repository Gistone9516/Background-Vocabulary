# C2.3 스펙 — 게이팅·한도 (buildflow ②, SoT §4에서 파생)

> 상위 정본 = `../인터페이스계약-v2.md` §4. v1 이월 = `adapters/workers/src/usage-counter.ts`(Upstash INCR/EXPIRE) + index.ts 게이팅 헬퍼.

## 0. 범위 / 비범위
- **범위**: 비용 엔드포인트 공통 미들웨어 — 티어 해석(JWT)·IP 분/일 리밋·전역 일일 캡·고위험 1차 방어(NFR-306)·free 주간 한도(TR-02 재차감 금지)·pro 전용(402)·상세 열람 한도·GET /usage.
- **비범위(이월)**: clientCheck(웹 Origin 화이트리스트·데스크톱 클라 토큰, NFR-305) = C4 스펙에서 확정(로컬 skip). 실 Upstash 카운터 = C2.4(포트만 정의, 로컬은 인메모리).

## 1. 핵심 설계 결정
### 1-1. CounterStore 포트 하나로 통합
- v1의 UpstashCounter·UsageCounter·GlobalDailyCap은 전부 `INCR + 최초 EXPIRE / GET` 패턴(키 스킴만 다름). → 단일 `CounterStore`(hit/get) 포트로 통합. 구현 = 인메모리(local·C2.3) / Upstash REST(C2.4). 키는 미들웨어가 조립(`ip:{ip}:min|day`·`global:day`·`week:{user|anon}:{id}`·`detail:{uid}:{sid}`).
### 1-2. 게이팅은 http-app 미들웨어(§4 "공통 미들웨어")
- 순수 판정(고위험 정적 키워드)은 `core/gating`(isHighRiskInput). 카운터·티어 의존은 http-app 미들웨어(`installGating`, 라우트 등록 전 install). 티어는 미들웨어가 `c.set("tier")`로 심고 파이프라인 라우트가 우선 사용(바디 tier는 게이팅 미적용 mock 부트 폴백).
### 1-3. TR-02 재차감 금지(구조적 강제)
- free 주간 차감은 **/classify(신규 탐색 시작)에서만**. 세션 재개(PUT /sessions + /next 루프)는 /classify를 호출하지 않으므로 재차감이 구조적으로 불가능. 선-확인(get) 후 통과 시 hit(재시도마다 증가 방지, v1 교훈).
### 1-4. fail-open(NFR-404)
- 카운터 장애 시 게이트는 통과(서비스 우선). 리밋·캡·주간·상세 전부 try/catch로 fail-open.
### 1-5. 고위험 1차 방어 보수성
- 배경 어휘 학습 도구라 도메인 용어(암·종양 등)는 막지 않고 "개인 위해"(자해·개인 진단/법적 판단 요청) 신호만 정규식으로 차단. 정밀 판정은 LLM 2차(domain_risk, core 파이프라인).

## 2. 엔드포인트 영향
- 비용 엔드포인트(classify·next·preview·relate·recommend·detail·summarize·refine-primer): IP/전역 캡 + 티어 심기.
- classify·recommend 입구: 고위험 403(HIGH_RISK_REFUSED).
- classify: free 주간 402(WEEKLY_LIMIT).
- summarize·refine-primer: free 402(PRO_ONLY).
- detail: free 세션당 402(DETAIL_LIMIT).
- GET /usage: { tier, anonymous, weeklyRemaining }.

## 3. 로컬 검증 (완료 기준)
- e2e-gate 9/9: 주간한도 소진(2회 200→3회 402)·/usage 잔여 0·고위험 403·pro 전용 402·paid 업그레이드(DB tier=paid + refresh)로 우회. pg·auth·mock 게이트 무회귀.

## 4. 열린 항목
- clientCheck(Origin·데스크톱 토큰) = C4. 실 Upstash CounterStore = C2.4. 익명 키 정밀화(설치 UUID vs IP)는 실측 후 조정.
