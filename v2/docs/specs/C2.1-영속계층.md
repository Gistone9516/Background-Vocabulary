## 1. 핵심 설계 결정

### 1-1. SqlRunner 포트 — 드라이버는 갈아끼우고 리포는 한 벌 (DRY, 적층 방지)
- **문제**: local = node-postgres, aws = RDS Data API. 리포를 계층마다 복제하면 같은 SQL이 두 벌이 되어 스키마 변경 시 이중 유지보수(누더기).
- **결정**: 얇은 **`SqlRunner` 포트**(`query`/`execute`/`transaction`) 뒤로 드라이버를 감춘다. 리포 구현은 SqlRunner에만 의존해 **한 벌만** 작성한다. local이 `PgSqlRunner`(pg), aws가 `DataApiSqlRunner`(C2.5)를 주입한다. SQL은 PostgreSQL 방언 정본(§0-2).
- **Data API 제약 흡수 지점(§6)**: `transaction()`은 3분 상한을 전제로 짧게만 설계(장기 로직에 트랜잭션 걸치기 금지 = 리포 규칙). 목록은 전부 커서 페이지네이션(1MB 상한). 이 규칙은 리포가 지키고, SqlRunner는 드라이버 차이(파라미터 바인딩·행 매핑)만 흡수한다.

### 1-2. JSONB 매핑 경계
- `narrow·generated·primer·term·domain_tags·job_type·gap_type` = JSONB. 리포가 **도메인 타입 ↔ JSONB 직렬화**를 경계에서 전담(리포 밖으로 SQL·직렬화 누수 금지, 코드규약 §3 리포지토리). 조회 축(user·project·시간·term_norm)만 컬럼·인덱스.

### 1-3. 커서 페이지네이션 (1MB 상한 대응)
- `GET /sessions·/assets` = keyset 커서(`(updated_at|created_at, id)` 기반). 목록 응답은 **요약 필드만** — `term`·`narrow`·`generated` 등 대형 JSONB는 목록에서 제외(단건 조회로만). 커서 = 불투명 base64(sortKey,id).

### 1-4. 소유권·소프트삭제·승격 (SoT §3-3 검토 정정 반영)
- `PUT /sessions/:id`: 대상 session_id가 **타 user_id 소유면 409**(횡령 방지). 익명→로그인 승격은 서버가 **새 session_id 발급**(클라 임시 id는 client_ref로만 기록).
- `DELETE`=소프트(`deleted_at`), `POST …/restore`=유예 내 복구(경과 후 404). 목록 기본 `deleted_at IS NULL`.
- `DELETE /projects/:id`: 세션·자산의 소속만 해제(FK `ON DELETE SET NULL`), 세션 자체는 보존(v1 규칙 승계).

## 2. 패키지 변경 (파티션 맵 확장)
- **[신규] `packages/adapters/persistence/`** (`@vock/persistence`): `migrations/0001_init.sql`(§6 DDL) · `migrate.ts`(순번 적용 러너, SqlRunner 경유) · `repositories/{session,asset,knowledge,project}-repo.ts`(SqlRunner 의존, 포트 구현) · 배럴. **shared에만 의존**.
- **[shared]** `ports/persistence-ports.ts`(신규): `SqlRunner`·`SessionRepository`·`AssetRepository`·`KnowledgeRepository`·`ProjectRepository` + 커서·페이지 타입. `types/persistence.ts`(신규): `SessionRec·NarrowSnap·AssetTerm·KnowledgeState·Project·PrimerDoc`(SoT §2-3 이식).
- **[http-app]** `createApp` deps 확장 → `AppDeps`(파이프라인 포트 + 리포 4종). `routes/crud-routes.ts`(신규): `/sessions*·/assets·/projects·/knowledge`. 인증 미들웨어는 자리만(§4), C2.1은 `x-user-id`를 **DEV_MODE 한정** 수용(명시 TODO: C2.2에서 JWT 파생 user_id로 교체).
- **[local]** `PgSqlRunner`(`pg`) + `buildLocalDeps`(mock LLM + 실 PG 리포 조립) + 마이그레이션 적용. devDep `pg`·`@types/pg`.
- **[scripts]** `e2e-pg.mjs`: Docker PG 대상 CRUD 왕복(생성→조회→담기→소프트삭제→restore→소유권 409). 기존 mock e2e와 별개 게이트.
- **[루트]** `docker-compose.yml`(로컬 Postgres 16). `boundary-check` ALLOWED에 `persistence` 등록.

## 3. 의존 방향 (경계 게이트 갱신)
```
shared ← persistence ← local
shared ← core ← http-app ← local
```
- http-app은 **리포 포트(shared)** 에만 의존(persistence 직접 참조 금지 — 부트가 구현 주입). local이 persistence+core+http-app 조립.

## 4. 로컬 검증 (완료 기준)
1. `docker compose up -d` → Postgres 기동.
2. 마이그레이션 적용(`migrate.ts`).
3. `e2e-pg`: 세션 생성(PUT)→목록/단건 조회→담기(keep)→knowledge upsert→소프트삭제→restore→**타 소유 PUT 409**→프로젝트 삭제 시 세션 보존. 전부 PASS.
4. 기존 게이트(tsc -b·guard·boundary·size·parity·mock e2e) 무회귀.
