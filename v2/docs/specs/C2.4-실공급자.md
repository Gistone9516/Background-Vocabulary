# C2.4 스펙 — 실 공급자 어댑터 (buildflow ②, SoT §0-2·§5에서 파생)

> v1 이월 = `core/llm`(DeepSeek SSE)·`adapters/search-providers/{tavily,upstash-cache}`·`usage-counter`. 로직 verbatim, 런타임/scope만 교체.

## 0. 범위 / 비범위
- **범위**: `@vock/providers`에 실 공급자 4종 — DeepSeekLlmClient(LlmClient)·TavilySearchProvider(SearchProvider, ko 가드)·UpstashCacheStore(CacheStore)·UpstashCounterStore(CounterStore). 실 local 계층 조립(`buildLocalRealDeps`, .env 실키).
- **비범위(이월)**: 실키 스모크(DeepSeek/Tavily/Upstash 실호출) = 핸즈온(키 보유 시). aws Secrets 주입 = C2.5.

## 1. 핵심 설계 결정
### 1-1. 공급자 어댑터는 core가 아니라 providers
- v1은 DeepSeek 클라이언트를 core/llm에 뒀으나 공급자 특수성(DeepSeek 엔드포인트·SSE wire·thinking 플래그)이라 **§0-1 위반**. v2는 core에서 분리해 `@vock/providers`(웹표준 fetch, 런타임 무관, local·aws 공유)에 둔다. core 파이프라인은 주입된 LlmClient 포트만 안다.
### 1-2. DeepSeek 파일 분할(코드규약 300행)
- v1 단일 파일(~370행)을 `deepseek/client.ts`(DeepSeekLlmClient·parseLooseJson)와 `deepseek/sse-parser.ts`(consumeSseStream·증분 Term 추출)로 분할. 파서는 결정적 테스트를 위해 export.
### 1-3. CounterStore 통합 구현
- Upstash INCR/EXPIRE/GET을 C2.3의 단일 CounterStore 포트로 구현(UpstashCounterStore). v1의 3개 카운터 클래스를 대체.
### 1-4. Upstash 폴백
- `buildLocalRealDeps`는 upstash 미설정 시 인메모리 캐시·카운터로 폴백(실 LLM만으로도 로컬 개발 가능).

## 2. 검증 (완료 기준)
- **SSE 증분 파서 결정적 테스트(e2e-sse)**: 임의 청크 경계로 쪼갠 DeepSeek SSE를 흘려 완성 Term만 순서대로 emit, done 종료 — 네트워크 없이 최난도 로직 검증. 목 게이트에 편입.
- 나머지(Tavily·Upstash·DeepSeek 실호출)는 타입체크·구조 검토로 확인, 실키 스모크는 핸즈온.
- pg·auth·gate·mock 게이트 무회귀.

## 3. 열린 항목
- 실키 스모크(각 공급자 1회 실호출) = 핸즈온. Tavily ko 대체(네이버 등) = 보류(MVP). aws Secrets → cfg 주입 = C2.5.
