# v2 코드 규약 (사용자 지시 2026-07-20·21 — 첫 단추 규칙)

구현 전 확정하는 하드 규칙. 모든 구현자(opus·위임 에이전트)는 코드 작성 전 이 문서를 정독한다. 위반은 리뷰 반려 사유다.

## 1. 단일 파일 누적 절대 금지
- **파일 = 단일 책임 1개.** 화면 1개, 훅 1개, 리포지토리 1개, 라우트 그룹 1개. "일단 여기 넣자"를 금지한다.
- **크기 신호**: 소스 파일이 약 200행을 넘으면 분리를 검토하고, 300행은 분리 사유서(주석 아님, 리뷰에서 설명) 없이 넘을 수 없다. v1 App.tsx(904행 누적)가 반면교사다.
- 문서도 동일: 진행 이력을 한 문서에 append하지 않는다. 상태 문서는 "현재 상태"만 유지하고, 이력은 git이 담당한다.

## 2. 인덱싱 철저
- 각 패키지는 루트에 **역할·구조를 설명하는 짧은 README**(10~20행)와 공개 표면을 정의하는 **index.ts(배럴)**를 갖는다. 패키지 밖에서는 배럴 경유로만 import(딥 임포트 금지 — dependency-cruiser로 강제).
- `v2/docs/`는 `_index.md`가 문서 카탈로그의 정본이다. 문서 추가·폐기 시 _index 갱신이 커밋 조건.
- 새 화면·모듈을 추가하면 해당 패키지 README의 구조 맵을 같은 커밋에서 갱신한다.

## 3. 소프트웨어 공학 패턴 (강제 지점)
| 패턴 | 적용 지점 | 근거 |
|---|---|---|
| 포트-어댑터(헥사고날) | 모든 외부 자원(LLM·검색·DB·캐시·플랫폼 API) — 포트는 shared, 구현은 adapters | SoT §0-1, 3계층 실행의 전제 |
| 의존성 주입 | core·http-app은 포트 구현을 생성하지 않고 부트에서 주입받는다 | 테스트 가능성, 계층 교체 |
| 리듀서(상태 기계) | ui-shared 화면 상태 — 액션 타입 명시, 화면별 리듀서 분리(단일 거대 리듀서 금지) | v1 useReducer 승계 + 누적 금지 |
| 리포지토리 | 영속 접근(Session/Asset/Knowledge/Project/UserRepository) — SQL은 리포 구현 밖으로 새지 않는다 | SoT §3-3 |
| 팩토리(부트 조립) | adapters/{aws,local}의 부트 = 포트 구현 조립 팩토리 하나로 집중 | 계층 분기의 유일 허용 지점 |
| 전략 | 모델 라우팅(위험도별)·프라이머 템플릿(로케일별) 등 분기 로직 | if-else 누적 방지 |
- 패턴을 위한 패턴은 금지: 위 표 밖에서는 "이 분기가 셋째로 늘었나?"가 패턴 도입 신호다.

## 4. 작성 스타일
- 주석은 코드가 못 보여주는 제약만(전역 규칙: 특수기호 금지, 사람이 쓰는 문장으로).
- 네이밍·구조는 기존 파일의 관례를 먼저 읽고 계승한다(스타일 분석 후 작성).
- 에러는 단계별로 잡아 어느 단계가 왜 실패했는지 남긴다(자동화 강건성 규칙 준용).

## 5. 게이트 연결
- C1 완료 기준에 본 규약 위반 0을 포함한다: 경계 게이트 3종 + 파일 크기 검사(스크립트) + 배럴 경유 import 검사.
- 위임 에이전트 프롬프트에는 이 문서 경로를 필수로 첨부한다.
