> |---|---|---|---|
> | `POST /auth/google` | `{ code, code_verifier, redirect_uri }` | `{ access_token, refresh_token, expires_in, user:{ email, tier } }` | Worker가 Google code 교환(client_secret) → id_token 검증 → email로 find/create(google_sub 링크) → access(15분)+refresh(30일) 발급. 실패=401 `AUTH_FAILED` |
> | `POST /auth/refresh` | `{ refresh_token }` | `{ access_token, refresh_token, expires_in }` | refresh 서명+exp 검증 → 블랙리스트(jti) 검사 → **D1에서 최신 tier 재조회** → access 재발급(+refresh 회전). 만료/폐기=401 `TOKEN_REVOKED`. 만료 2분 전 클라가 선제 호출 |
> | `POST /auth/logout` | (Bearer) | `{ ok:true }` | 클라 저장 삭제 + (옵션)블랙리스트 등록 |
> | `POST /checkout` | `{ currency }`(Bearer) | false: `{ status:"coming_soon" }` / true: `{ checkout_url, session_ref }` | `PAYMENT_ENABLED=false`면 스텁. true=Phase 1 어댑터 |
> | `GET /subscription/status` | (Bearer) | `{ tier, subscription_status, expires_at, access_token }` | D1 직접 조회 후 새 JWT 재발급(K5 폴링·복구) |

### B-7. 티어 판정 로직 (resolveTier 교체)
- 유형: 확정결정
- 출처: 배경어휘-사이드탭-인터페이스계약.md 8-6절, 280~291행
- 폼팩터: 무관(v2 그대로 유효)
- 원문: > ### 8-6. 티어 판정 (readTier 교체, §A4)
> ```
> resolveTier(req, env):                                                  // per-request = NETWORKLESS(외부호출 0)
>   if env.DEV_MODE && env.DEV_FORCE_TIER: return env.DEV_FORCE_TIER     // 개발 전용, prod에선 둘 다 미설정
>   claims = verifyAccessJwt(bearer(req), [JWT_SECRET_CURRENT, JWT_SECRET_PREV])  // 서명+exp+kid만(블랙리스트 조회 없음)
>   if claims: return claims.tier
>   return "free"                                                          // 무인증·익명 = free(기존 흐름 보존)
> ```
> - **블랙리스트는 per-request 아님 — refresh 시점에만 검사**(networkless 보존, §A4). access exp=15분이라 취소·환불 반영 지연 ≤15분(표준 short-exp 패턴). 즉시성 더 필요하면 Phase 1에서 exp 단축.
> - **x-tier 헤더 완전 폐기.** 현재 실 pro 사용자 0(미리보기 토글뿐)이라 **클린 컷오버**, 구버전 fallback 불요(K4 우려 해소).
> - 유료 엔드포인트(`/summarize`·`/detail`·`/recommend`·`/next`)는 이 `resolveTier` 결과로 게이팅(현 호출부 시그니처 유지, 내부만 교체).
> - `PAYMENT_ENABLED`는 티어 판정에 **영향 없음**(b안). checkout·CTA만 게이트.
