### 12c-3. c-0-1 진입 "조건"이 좁히기·추천에 전혀 반영되지 않음 [치명]
- 유형: 기능요구
- 출처: 평가-구현_비판검토.md c-0-1 (332~334행)
- 폼팩터: 무관(v2 그대로 유효)
- 원문: > **c-0-1 [치명] 진입 시 입력한 "조건"이 좁히기·추천에 전혀 반영되지 않는다 (H1 가시성, H2 일치).**
진입 화면의 `+ 조건` 토글로 받는 `state.cond`는 `submitEntry → startNarrow(v)` 경로에서 `classify`에도 `runRecommend`에도 전달되지 않는다. `cond`가 쓰이는 곳은 마지막 담은 화면의 `buildSummary`(`sum_cond`) 한 곳뿐이다. 즉 사용자가 진입에서 "나는 비전공자 기획자고 마케팅 관점이 필요해" 같은 조건을 적어도, 그 조건은 질문 분기와 어휘 선정에 0의 영향을 주고 30초 뒤 붙여넣기 요약에서야 처음 등장한다. 사용자는 자신이 제공한 입력이 결과를 바꿨다고 합리적으로 기대하지만(시스템 상태 가시성 위반), 실제로는 무시된다. 제품의 핵심 가치인 "표현 면제·의도 결정화"와 정면으로 충돌하는 결함이다 — 가장 비싸게 표현한 입력이 가장 늦게, 가장 약하게 반영된다.
> - 개선: `classify`/`buildRecInput`에 `cond`를 `user_condition` 또는 `context_object`로 전달해 좁히기 첫 질문과 어휘 선정에 즉시 반영하거나, 그게 설계상 불가하면 조건 필드 자체를 "결과엔 직접 반영되지 않고 마지막 요약에만 들어가요"라고 명시해 기대를 교정한다. 전자가 정답이다. 현재 상태는 사용자를 기만하는 수준의 불일치다.

### 12c-4. c-0-2 홈 복귀가 전체 리셋이며 경고 없음
- 유형: 기능요구
- 출처: 평가-구현_비판검토.md c-0-2 (336~338행)
- 폼팩터: 재검토(폼팩터 전환으로 의미 변할 수 있음 — "로고=홈" 관용이 웹/데스크톱 내비게이션에서는 다르게 설계될 수 있음)
- 원문: > **c-0-2 홈 복귀가 전체 리셋이며 경고가 없다 (H3 제어·자유, H5 오류 예방, H9 복구).**
헤더 브랜드 버튼이 유일한 홈 진입점이고 `goHome`은 `dispatch({type:"reset"})` 즉 `initial()` 전체 초기화다. 담기 화면·어휘 화면에서 무심코 로고를 누르면 현재 세션 상태(열어둔 카드, 입력 중인 방향성, 좁히기 진행)가 통째로 날아간다. 담은 어휘 자체는 `persist`로 storage에 남지만, 진행 중이던 탐색 컨텍스트는 복구 불가다. 게다가 로고는 보통 "홈으로"라는 관용이라 사용자는 "데이터 초기화"를 예상하지 못한다(H2 위반).
> - 개선: 진행 중 탐색이 있을 때 로고 클릭에 확인 다이얼로그를 띄우거나, 최소한 어휘/담기 화면에 명시적 "처음으로(새 탐색)" 링크를 두고 로고는 비파괴적 "현재 탐색의 처음으로"로 분리한다.

### 12c-5. c-0-3 언어 선택이 진입 화면에서만 가능
- 유형: 제약
- 출처: 평가-구현_비판검토.md c-0-3 (340~342행)
- 폼팩터: 재검토(폼팩터 전환으로 의미 변할 수 있음)
- 원문: > **c-0-3 언어 선택이 진입 화면에서만 가능하다 (H7 유연성·효율, H3 제어).**
`Header`는 `state.screen === "entry"`일 때만 언어 셀렉터를 렌더한다(코드 주석: 이미 만들어진 LLM 텍스트가 언어 변경에 반응하지 않아 탐색 중 전환을 막음). 의도는 이해되지만, 결과적으로 사용자가 좁히기에 들어간 뒤 "아 영어로 볼걸" 하면 전체를 리셋(c-0-2)해야만 언어를 바꿀 수 있다. UI 문자열(버튼·라벨)은 언어 변경에 즉시 반응하므로, LLM 콘텐츠만 다음 호출부터 반영된다고 안내하면 탐색 중 전환을 굳이 막을 이유가 약하다.
> - 개선: 탐색 중에도 셀렉터를 노출하되, 변경 시 "새로 만드는 어휘부터 적용돼요" 같은 인라인 노트를 띄운다. 막더라도 진입에서 언어 선택이 더 눈에 띄게(현재 셀렉터는 우상단 작은 회색 박스) 배치해 첫 선택을 돕는다.

### 12c-6. c-0-4 터치/클릭 타깃이 WCAG 최소 기준 미달인 컨트롤 다수
- 유형: 비기능요구
- 출처: 평가-구현_비판검토.md c-0-4 (344~346행)
- 폼팩터: 무관(v2 그대로 유효)
- 원문: > **c-0-4 터치/클릭 타깃이 WCAG 최소 기준 미달인 컨트롤이 다수 (H4 표준, 접근성).**
WCAG 2.2 기준 타깃 최소 24x24px(AA), 권장 44x44px다. 코드상 다음이 미달이다: 예시 재추첨 `.shuffle` 22x22(`opacity:.3`), 히스토리 삭제 `.histdel` 20x20(`opacity:.3`), 첨부 제거 `.filechip button` 18x18, 헤더 도움말 `.help` 24x24(경계선), 담기 미니 토글 `.keepmini` 26x26. 특히 `.shuffle`·`.histdel`은 크기도 작고 평소 투명도 30%라 발견성과 적중률이 동시에 나쁘다(파인트 컨트롤 + 저대비 = 이중 페널티).
> - 개선: 시각 크기는 유지하되 클릭 가능 영역(padding 또는 `::before` 히트박스)을 최소 24px, 가능하면 44px로 확장한다. 평소 투명도 30%는 35~45% 이상으로 올리거나 호버 없이도 식별되게 한다. (참고: 재인>회상과 활성화 에너지 감소 원리상, 보조 기능이라도 "보여야 눌린다" — Fogg Behavior Model의 Ability 레버, M 신뢰도.)

### 12c-7. c-0-5 본문 폰트 일부가 과소
- 유형: 비기능요구
- 출처: 평가-구현_비판검토.md c-0-5 (348~350행)
- 폼팩터: 재검토(폼팩터 전환으로 의미 변할 수 있음 — 근거가 "사이드패널 최대 480px" 폭 제약)
- 원문: > **c-0-5 본문 폰트 일부가 과소 (H8 가독성, 접근성).**
`.note`/`.entryNote` 11.5px, 예시 칩 `.sg` 10.5px, `.plan` 배지 11.5px. 사이드패널은 본문 폭이 좁아(최대 480px, 실패널은 더 좁을 수 있음) 10.5px 텍스트는 가독 하한에 가깝다. 특히 진입 화면 예시 칩은 "재인으로 진입을 돕는다"는 핵심 장치인데 글자가 가장 작다 — 기능 중요도와 시각 위계가 역전됐다.
> - 개선: 예시 칩 본문은 최소 12~13px로 키운다. 안내성 `.note`는 11.5px 유지 가능하나, 사용자가 읽어야 하는 정보(고위험 분야 안내 등)와 단순 각주를 구분해 전자는 키운다.

### 12c-9. c-0-7 스트리밍 중 빈 상태·취소 수단 부재

- 폼팩터: 무관(v2 그대로 유효)

> - 개선: 추천 첫 카드 전까지 정의된 `.skel` 스켈레톤 2~3개를 띄워 "오는 중"을 형태로 보여준다. 예상 개수(`Max N`)를 진행 표시에 활용한다. 길어질 때 "중단" 링크를 노출한다.

### 12c-11. c-1-2 예시 칩 "다른 예시" 버튼의 발견성 낮음
- 유형: 기능요구
- 출처: 평가-구현_비판검토.md c-1-2 (368~370행)

`.shuffle`은 칩 줄 끝 22x22 아이콘에 평소 투명도 30%다. 예시 칩은 이 제품의 표현-면제 진입을 여는 핵심 장치(재인으로 시작점 제공)인데, "이게 전부가 아니라 새로 뽑을 수 있다"는 사실 자체가 거의 숨겨져 있다. 회전 화살표 아이콘만으로는 "예시 더 보기/다시 뽑기"라는 의미가 즉시 읽히지 않는다(H2).
> - 개선: 평소 투명도를 올리고, 칩 풀이 더 있음을 텍스트("다른 예시 ↻")로 명시하거나 칩 영역 하단에 작은 라벨을 둔다. 첫 방문자에게는 예시 칩이 가장 중요한 진입로이므로 위계를 높인다.

### 12c-12. c-1-3 첨부 기능이 무료에게 안 보이다가 드롭하면 페이월
- 유형: 기능요구

- 유형: 기능요구
- 출처: 평가-구현_비판검토.md c-4-3 (440~442행)
- 폼팩터: 무관(v2 그대로 유효)
- 원문: > **c-4-3 "확인된 출처 없음"의 톤이 신뢰를 흔들 수 있다 (H2).**
`detail_nosrc`("확인된 출처 없음")는 정직한 처리지만(좋음), 같은 화면에서 LLM이 생성한 개념/활용은 단정적으로 제시된다. 출처 없는 카드에서 사용자는 "그럼 이 내용은 믿어도 되나"를 판단할 근거가 약하다. (참고: LLM은 sycophancy로 사용자 프레임을 확증하는 경향이 실증됐다 — Sharma et al., ICLR 2024, H. 출처 부재 시 내용 신뢰는 더 신중해야 한다.)
> - 개선: 출처 없는 항목에는 "일반적 설명이며 출처 미확인" 같은 약한 면책을 일관되게 두되, 과하지 않게. `entry_note`의 "AI가 만든 결과예요" 면책이 상세 단위에서도 가볍게 반복되면 좋다.

### 12c-27. c-5-1 "담은 어휘 보기" 버튼이 0개일 때 비활성인데 이유 없음
- 유형: 기능요구
- 출처: 평가-구현_비판검토.md c-5-1 (448~450행)
- 폼팩터: 무관(v2 그대로 유효)
- 원문: > **c-5-1 "담은 어휘 보기" 버튼이 0개일 때 비활성인데 이유가 없다 (H1, H10).**
`Terms` 푸터의 `kept_view` 버튼은 `keptCount === 0`이면 `disabled`다. 비활성 사유(담은 게 없음)가 버튼 자체엔 안 적혀 있어, 사용자는 "왜 안 눌리지"만 겪는다. 옆 카운트("0개 담음")가 단서이긴 하나 연결이 약하다.
> - 개선: 비활성 버튼에 "어휘를 먼저 담아주세요" 툴팁을 달거나, 0개일 때 버튼 라벨을 "담은 어휘 없음"으로 바꿔 상태를 자명하게 한다.

### 12c-28. c-5-2 "메인 AI에 붙여넣기" 요약의 가치가 화면에서 충분히 안 팔림
- 유형: 기능요구
- 출처: 평가-구현_비판검토.md c-5-2 (452~454행)
- 폼팩터: 무관(v2 그대로 유효)
- 원문: > **c-5-2 "메인 AI에 붙여넣기" 요약의 가치가 화면에서 충분히 안 팔린다 (H1, H8).**
이 화면의 핵심 산출물은 `buildSummary`로 조립된 붙여넣기 텍스트(분야·조건·어휘·맥락·요청 자동조립)다. 이게 제품의 "의도 결정화 → 메인 AI로 이관" 가치를 실현하는 지점인데, `.summary` 블록은 평범한 회색 박스이고 복사 버튼은 우측 컴팩트 `.pillbtn`이다. 가장 중요한 행동(복사해서 메인 AI로)이 시각적으로 가장 강조돼야 하는데 위계가 낮다.
> - 개선: 복사 버튼을 주 행동으로 키우고("이 정리를 메인 AI에 붙여넣기"), 복사 성공 후 "이제 ChatGPT/Claude에 붙여넣으세요" 같은 다음 행동 안내를 붙인다. 현재 `terms_foot_note`에 비슷한 안내가 있으나 어휘 화면에 있고 정작 복사 시점엔 없다.

### 12c-29. c-5-3 "AI로 더 정리"(pro)의 잠금 표시가 버튼 안에만 있음
- 유형: 기능요구

무료에서 `aiRefine`은 페이월로 보낸다. 버튼 라벨이 `refine_locked` + 자물쇠로 잠금을 표시하긴 하나(좋음), 옆 방향성 입력란(`.field`)은 무료도 자유롭게 타이핑할 수 있다. 사용자가 방향성을 길게 적은 뒤 버튼을 누르면 페이월 → 입력이 헛수고가 된다.
> - 개선: 무료일 때 입력란도 함께 잠그거나, 입력 위에 "AI 정리는 pro 기능"을 미리 표시해 헛입력을 막는다.

### 12c-30. c-5-4 뒤로 가기 라벨이 맥락(histView)에 따라 달라지는데 위치는 같음
- 유형: 기능요구
- 출처: 평가-구현_비판검토.md c-5-4 (460~462행)
- 폼팩터: 무관(v2 그대로 유효)

좌상단 링크가 `histView`면 "← 처음으로", 아니면 "← 어휘로 돌아가기"로 바뀐다. 같은 자리·같은 외형의 링크가 상황에 따라 다른 목적지로 간다. 히스토리에서 열었는지 방금 만들었는지를 사용자가 항상 기억하진 않으므로, 같은 버튼이 어디론가 다르게 데려가면 혼란이 생긴다.
> - 개선: 둘 다 명시적 목적지를 라벨에 담고 있으니(그 자체는 양호), 추가로 화면 상단에 현재 맥락("이전 탐색 다시 보기")을 표시해 어디서 왔는지를 알린다.

### 12c-32. c-6-2 페이월 진입 사유가 항상 명확하진 않음
- 유형: 기능요구

- 폼팩터: 무관(v2 그대로 유효)

페이월은 여러 경로(추천 한도 소진, 첨부 시도, 더보기, 상세 한도, 검색 재탐색, AI 정리)에서 호출되는데, `limitHit`이 true일 때만 상단 콜아웃(`pw_limit`)이 뜬다. 첨부/더보기 등으로 온 경우는 `limitHit:false`라 "왜 여기로 왔는지" 사유 표시가 없다. 사용자는 자기 행동과 페이월의 인과를 잃는다.

### 12c-33. c-7-1 튜토리얼 4스텝 강제 노출 + 본문이 설득문

첫 방문 시 자동으로 4스텝 모달이 뜬다(`tutorialOpen`). 스킵(건너뛰기) 버튼이 없어 백드롭 클릭이나 마지막까지 진행해야 닫힌다. 빨리 써보고 싶은 사용자에게는 마찰이다. 또 목적·이유 3스텝은 제품 철학(왜 만들었나)을 설득하는 내용인데, 첫 진입자 다수는 "왜"보다 "어떻게"가 급하다.

검증 범위: 원본 v1/docs/평가-구현_비판검토.md 321~480행(끝) 전문을 Read로 재독한 뒤 본 원장(12c-1~12c-33) 33개 항목과 대조.

- 누락 검사: 321~480행 전 라인이 33개 항목 중 하나 이상의 "출처" 인용 범위에 포함되는지 라인 단위로 전수 대조함. 커버되지 않은 라인 0건. 절 구조(c-0~c-7, 각 하위 c-x-y)와 개별 요구·수치·제약·기각대안 전부가 12c-1~12c-33에 대응됨을 확인. 추가로 발견된 누락 항목 없음.
- 요약화 검사: 각 항목의 "원문:" 인용 블록을 원본 해당 행과 문자열 단위(공백 정규화 후 부분일치 + 항목별 특수문자(em dash "—", 가운뎃점 "·") 개수 대조)로 전수 비교함. 33개 항목 전부 원문과 정확히 일치(의역·축약·누락 문자 없음). 볼드 마크업(`**...**`), 코드 인용(백틱), 괄호주(예: Fogg Behavior Model, Sharma et al. ICLR 2024), 수치(24x24/44x44px, 22x22/20x20/18x18/26x26, 2.4초/3초, MIN_Q~maxQ 3~8턴 등) 전부 원문 그대로 보존됨을 확인.
- 경계 항목 12c-33(c-7-1)은 원본상 480행에서 시작해 482행까지 이어지는 항목이라, 원장이 이미 481~482행까지 포함해 전문을 수록함(원장 자체 명시). 단, 같은 c-7 절의 다음 항목 c-7-2(원본 484~486행, 모달 접근성/포커스 트랩·ESC 관련)는 배정 범위(321~480행) 밖이므로 이 원장(12-평가비판-c)에는 포함하지 않음 — 별도 청크(v1-승계원장에 12-평가비판-d 파일 없음, 현재 미배정 상태로 확인됨) 소관.
