### A-3. 존재이유 방어 명제 (deepflow C1 격상)
- 유형: 확정결정
- 출처: 배경어휘-사이드탭-기획-구체화.md 24~32행 (### 1-1)
- 폼팩터: 무관(v2 그대로 유효)
- 원문:
> ### 1-1. 존재이유 방어 (가장 중요, deepflow C1에서 격상)
>
> 비판 패널 세 렌즈(사후부검, 제1원리, UX)가 공통으로 친 급소: **"메인 AI에게 그냥 물어보면 되는데 왜 이 탭을 먼저 여나?"** 이건 열어둬도 되는 질문(원래 §9)이 아니라 제품 생사를 가르는 전제이므로 여기서 답을 못 박는다.
>
> - **방어 명제:** 메인 AI의 되묻기(clarifying question)는 사용자가 *무엇을 물어야 할지 어느 정도 아는* 상태를 전제로 작동한다. 이 서비스는 **"무엇을 모르는지조차 모르는" 상태**에서 출발해, 분야 어휘 지형을 비선형으로 펼쳐 클릭만으로 의중을 외화한다. 선형 대화로는 "내가 뭘 모르는지"가 드러나기 전까지 좁혀지지 않는다.
> - **부차 차별:** 대화창 바깥의 독립 준비 공간이라는 위상, 어휘를 한눈에 보는 시각 탐색 경험.
> - **종착점 교정 (2026-06-24 사용자 directive):** 이 제품의 종착점은 "복사용 프롬프트 생성"이 아니라 **사용자가 분야 개념을 이해해 스스로 말할 수 있게 되는 것**이다(핵심 표면 = 화면2 상세 열람·드롭다운). 복사·공유(화면3)는 그 이해를 선택적으로 가져가는 편의일 뿐 목적이 아니다. 측정 편의 때문에 paste를 성공 지표로 박았던 것이 정체성을 덮었던 것을 바로잡음(§7-1 반영).
> - **이 명제는 아직 가정이다.** MVP에서 직접 검증한다(§7-1의 존재이유 가설). 깨지면 제품 방향 재고. 기획 단계인 지금은 검증하지 않고 검증 설계만 박아둔다.
> - **흡수 리스크 동반:** 같은 공백을 메인 AI가 네이티브로 메우면 차별점이 소멸할 수 있음(§7-3 리스크).

> **[2026-06-22 실측 갱신]** 프롬프트3을 flash/pro로 실측(`정확도-최종-결과.md`): ① JSON mode 지원 확인 ② 한국어 전문어휘 품질 = pro 평균 ~77/100·flash ~67(고위험 도메인 오류·할루시네이션 잔존) ③ latency = pro ~51초·flash ~34초(thinking ON, 한국 TTFT 1.5초 우려 잔존) ④ **모델은 pro 채택**(out-of-sample 15% 기준은 pro만 충족). 단가 정밀·캐싱·대규모 부하는 여전히 0단계 선결. **공통 최신성 오류(간이과세 구기준 등)는 모델 컷오프 문제라 RAG·경고로 보완 필요(§7-3).** **[0단계 격자 실측]** thinking OFF는 TTFT 0.8초로 속도는 통과하나 정확도 급락(no-RAG flash 53·pro 59), **RAG 주입 시 +25~27점·최신성 100% 교정**. 정확도 85 목표 통과는 **pro + thinking OFF + RAG(85.4)** 뿐 → 채택 레시피(+ term 단위 스트리밍). flash+RAG(78)는 저위험 폴백. 상세 `0단계-실측-결과.md`.

- 원문:
> ## 2. 핵심 작동 루프 (MVP, 상세화)
>
> 원본 §7 루프를 화면 단위로 풀어쓰되 **입력 구조는 종합안(다지선다 엔진 + 어휘 태깅, 사용자 확정)**을 따른다. 각 화면의 실제 UI 문구 초안까지 박는다(구현자 임의결정 방지). 문구는 잠정안, 관찰로 다듬음.
>
> > **입력 구조 결정 (사용자 확정):** 의중을 좁히는 본 루프는 **내용을 담은 3~4지선다**다. "어휘를 보고 아는가"가 아니라 "이 상황 중 무엇이 당신에 가까운가"를 고르게 한다. 재인(recognition) 기반이라 비전문가의 메타인지 부담이 작고, 정보이론 분기(원본 §4)와 정합한다. [알아/몰라]는 마지막에 **좁혀진 핵심어 묶음을 태깅**하는 보조 단계로만 쓴다. 근거는 §3.
>
> > **MVP 범위 (작업한계 없음 반영):** 실시간 적응형 분기(매 선택마다 다음 선택지를 LLM이 생성)를 1차에 포함한다. 고정 선택지만으론 "클릭으로 의중이 좁혀지는가"를 검증 못 함. 실시간에 따르는 견고성 장치(JSON 계약 §5-1, pending 잠금 §3-1, 선택지 품질 가드·종료조건 §3-2, 지연 fallback §7-3)를 함께 구현. 1차는 LLM-prompted, 엔트로피 실제 계산(greedy EIG)은 2차.

### A-14. 3-2 선택지 생성 1차/2차 구분·수렴종료조건·depth cap
- 유형: 기능요구
- 출처: 배경어휘-사이드탭-기획-구체화.md 115~119행
- 폼팩터: 무관(v2 그대로 유효)
- 원문:
> ### 3-2. 선택지 생성 — 1차/2차 구분 + 운영 규칙
> - **1차(MVP): LLM-prompted 선택지 생성 (EIG 모사).** 매 선택 누적을 LLM에 넣어 다음 3~4개 내용형 선택지를 실시간 생성하되, 엔트로피를 실제 계산하지 않고 "의중을 가장 크게 가르는 선택지를 내놓으라"고 지시하는 휴리스틱이다. EIG라는 이름을 붙이지 않는다(과장 방지). 1차에서 볼 것은 "LLM 선택지가 의중을 좁히는 데 쓸 만한가"이지 "EIG가 도는가"가 아니다.
> - **2차: 진짜 greedy EIG.** 동작은 같되 누적 로그로 후보 분포 엔트로피를 *실제 계산*해 가장 크게 줄이는 분기를 고른다(LLM 휴리스틱에서 정보이론 최적화로). 1차와 2차 기준선을 여기서 분리.
> - **수렴·종료 조건 (적응형 필수, deepflow C3):** 실시간 분기는 "선택지 소진"이 없으므로 시스템이 끝을 정해야 한다. (a) 누적 선택 N회 초과(예: 6~8), (b) [더 깊이] 누적 k회 도달(예: 3), (c) 같은 도메인 태그 연속 j회(예: 3) 중 하나면 자동으로 정리 화면 유도("이 정도면 충분해 보입니다, 정리 볼까요?"). 수치는 §7-2 관찰로 보정. ([지금 정리 보기]는 사용자가 언제든 누르는 탈출구이고, 자동 유도는 종료 조건 충족 시 그 버튼을 강조하는 것)
> - **[더 깊이] depth cap (deepflow C3):** 세션 내 누적 상한(예: 3회). 도달 시 버튼 비활성화 + "정리 먼저 보고 새로 파보세요" 안내.

### A-19. 3-2 조건 조기수집(fast-input) — 시점 역전 해소
- 유형: 기능요구
- 출처: 배경어휘-사이드탭-기획-구체화.md 136행
- 폼팩터: 무관(v2 그대로 유효)
- 원문:
> - **조건 조기 수집 — fast-input (P16·P17, 시점 역전 해소):** user_condition·context_object는 화면2 추천 시점에 있어야 priority가 개인화된다. 그래서 ① 화면0 보조 입력(분야명·자유문장 외에 "관련 조건을 한 줄로" 선택 입력)과 ② 붙여넣은 원문(context_object)을 화면0~1에서 받아 프롬프트2·3에 함께 전달한다. [내 상황 추가](화면3)는 마지막 보강 채널로 유지하되, 추천 개인화의 1차 입력은 화면0~1에서 받는다. §3-1 context_object 이월 선언과 §5-1 계약을 여기서 일치시킨다.

### A-20. 3-2 조건 필수 판단 — 조건 의존도 기반(화이트리스트 금지)
- 유형: 확정결정
- 출처: 배경어휘-사이드탭-기획-구체화.md 137행
- 폼팩터: 무관(v2 그대로 유효)
- 원문:
> - **조건 필수 판단 (P17·P29, 조건 의존도 기반):** 고정된 도메인 목록이 아니라 "user_condition 없이는 추천·정리가 일반론에 그치는가"라는 **조건 의존도**로 판단한다(목록 하드코딩은 커버리지 누락을 부른다, P29). 프롬프트1이 의존도가 높다고 보면 화면0~1 조건 입력을 "선택"에서 "권장"으로 승격하고 케이스별 핵심 변수를 힌트로 제시한다(예: 거주 기간, 보유 수치, 소득, 건물 속성, 공시 범위, 검사 항목). 조건 의존 高 예시: 비교·판단·진단·세무·상속·법률·의료·ESG 공시·교육 행정·건축 행정·사회보험. **이 예시는 프롬프트 설계 시 few-shot 참고로만 쓰고 시스템 프롬프트에 화이트리스트로 나열하지 않는다**(나열하면 목록 외 조건 의존 도메인이 누락됨, P30). 판정 결과는 프롬프트1이 condition_required(bool)로 출력하고 프론트가 그 플래그로 승격한다(§5-1). 입력 마찰과 정확도의 트레이드오프는 §7-2에서 관찰한다.

### A-21. 3-2 user_condition 병합·재추천 규칙
- 유형: 확정결정
- 출처: 배경어휘-사이드탭-기획-구체화.md 138행
- 폼팩터: 무관(v2 그대로 유효)
- 원문:
> - **user_condition 병합·재추천 규칙 (P26):** user_condition은 두 채널에서 들어온다 — 화면0~1 보조입력(조기, 추천 개인화용)과 화면3 [내 상황 추가](보강·확정). **병합 우선순위: 화면3 최종값이 화면0 값을 덮는다**(마지막에 확인·수정한 값이 진실). 화면0 값은 화면3에 prefill로 보여 보강만 받고, 프롬프트4에는 이 병합 최종값을 넣는다. **재추천 트리거:** 조건을 화면0~1에서 못 받고 화면3에서 처음 받았거나 핵심 변수가 크게 바뀌면 화면2 추천은 이미 일반론으로 생성된 상태다. 이때 자동 재호출하지 않고(비용·지연) "바뀐 상황으로 추천 다시 볼래요?" 버튼을 제시해 누르면 프롬프트3을 재호출, 안 누르면 paste_text(프롬프트4)에만 반영한다.

### A-22. 3-3 데이터 단위 — Choice/Term JSON 계약
- 유형: 기능요구
- 출처: 배경어휘-사이드탭-기획-구체화.md 140~163행 (### 3-3 도입부 + 코드블록)
- 폼팩터: 무관(v2 그대로 유효 — 데이터 모델 자체는 폼팩터 독립)
- 원문:
> ### 3-3. 데이터 단위 (선택지 / 어휘 카드)
> 좁히기 엔진이 다루는 **선택지(Choice)**와 마지막 태깅 단계의 **어휘 카드(Term)**를 구분한다. JSON 계약은 §5-1.
> ```
> Choice {                          // 화면1, 다지선다 한 항목
>   label:       "가격이 왜 그렇게 매겨지는지"   // 일상어 상황문 필수(P6). 전문어 금지
>   domain_tags: [미시경제, 가격이론]            // 전문어는 여기에만. 의중 분포·중복 가드용
>   deeper:      true                            // [이 갈래 더 깊이] 가능 여부
> }
>
> Term {                            // 화면2, 우선순위 추천 어휘 카드(P13)
>   term:         "한계효용"
>   kind:         개념 | 방법론 | 지표 | 제도·규정 | 도구 | 절차 | 역할   // 단어의 종류(P14)
>   group:        "학습 안정화"                  // 상위 주제(클러스터). 그룹핑·지도 보기 조직 단위. 필수(P32)
>   priority:     1                              // 이 상황 적합 우선순위(추천 핵심)
>   why:          "가격 근거를 따지려는 당신 목적에 직접 쓰임"  // 왜 우선인지, 근거 필수
>   one_line:     "한 단위 더 소비할 때 늘어나는 만족"
>   tag:          미확인 | 알아 | 몰라 | 적용모름   // 알아=추천서 빼기, 몰라/적용모름=집중
>   direction:    "수치 높을수록 유리"            // 선택, 판단·협상·제도 맥락(P7)
>   use_example:  "가격 인하 정당화: 한계효용 체감으로 추가 할인의 매출 증가폭이 준다"  // 선택, 글쓰기(P12)
>   context_note: "당신 상황에선 이렇게 작동"      // 선택, 적용 한 줄(P11)
>   relates_to:   ["고정비"]                       // 선택, vocab_gap 유형 b. 이 어휘와 묶어 봐야 하는 다른 term (관계 표현, P19)
>   order:        2                                // 선택, 유형 b. 개념·절차의 선후 순서 (작으면 먼저, P19)
>   gap_type:     ["c"]                            // 선택, 이 어휘의 막힘 유형. 세션 기본값을 어휘별로 오버라이드(P27). a~e, 복합이면 배열
> }
> ```

### A-23. 3-3 vocab_gap 유형 분기 및 Term 필드 고정 매핑
- 유형: 확정결정
- 출처: 배경어휘-사이드탭-기획-구체화.md 165행
- 폼팩터: 무관(v2 그대로 유효)
- 원문:
> - **vocab_gap 유형 분기(P7):** 막힘은 (a)뜻 모름 (b)개념 관계·순서 (c)재번역(기존 언어에서 전문 언어로) (d)수치·결과 해석 (e)조항 상호작용으로 나뉜다. **유형 다음 Term 필드 매핑(고정, P27): a는 one_line, b는 relates_to·order, c는 use_example(재번역 예시)와 direction, d는 direction·context_note(수치 해석), e는 context_note·direction(상호작용·유불리).** "연결 문장"을 why·context_note에 비정형으로 구겨넣지 않는다(P19). **Term.gap_type가 있으면 그 어휘는 Term.gap_type만 적용한다(세션 기본값과 합집합이 아니라 대체, P30).**

### A-24. 3-3 gap_type 결정 주체·시점 (닭-달걀 해소)
- 유형: 확정결정
- 출처: 배경어휘-사이드탭-기획-구체화.md 166행
- 폼팩터: 무관(v2 그대로 유효)
- 원문:
> - **gap_type 결정 주체·시점 (P27, 닭-달걀 해소):** gap_type은 화면2 추천(프롬프트3)의 입력이라 그 전에 정해져야 한다. 결정 흐름은 **프롬프트1·2가 진입 자유문장·선택 이력·context_object에서 session 기본 gap_type을 LLM 추론**해 프롬프트3에 넘긴다(사용자 [몰라/적용모름] 태깅은 화면2 이후라 늦으므로 추론이 1차). 어휘마다 막힘이 다르면 Term.gap_type로 어휘별 오버라이드(세션 기본값 위에 얹음). 복합이면 배열(예: [c,d], P21). use_example은 글쓰기 job_type **또는 gap_type에 c(재번역) 포함** 시 생성한다(재번역이 핵심인 비글쓰기 시나리오 누락 방지, P27).
